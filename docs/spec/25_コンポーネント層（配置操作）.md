# コンポーネント層（配置操作）

## 目的

Slot に登録されたコンポーネントの位置を編集する API（配置操作）を定める。対象は `insertBeforeComponent` / `insertAfterComponent` / `removeComponent` / `replaceComponent` の 4 操作と、それを下支えする registry / collector 型の設計根拠である。

### スコープ

この文書で扱うもの:

- 配置操作の一覧と関数命名
- Core / Customize の役割分担
- `replaceComponent` の位置づけ
- registry / collector 型を採る設計根拠（Schema Builder 型を採らない理由を含む）

この文書で扱わないもの:

- 基本構造（JSX + Slot + `registerComponents`）と `addComponent` シグネチャ、Slot id / Component id の命名規則 — doc24

- 振る舞いの変更（Action パイプライン操作）— doc28（Core 側）/ doc30（Customize 側）
- ディレクトリ物理配置 — doc22

## 配置操作

### 操作一覧

| 操作 | 関数 | 主な使い手 |
|---|---|---|
| 追加 | `addComponent` | Core（標準構成を積み上げる）|
| 前挿入 | `insertBeforeComponent` | Customize（既存アンカーの前に差し込む）|
| 後挿入 | `insertAfterComponent` | Customize（既存アンカーの後に差し込む）|
| 削除 | `removeComponent` | Customize（不要な項目を除去）|
| 入れ替え | `replaceComponent` | Customize（位置を維持して中身を差し替え）|

`addComponent` のシグネチャと第3引数 props のマージ機構は doc24（基本構造）と doc26（Core 側）3.1 で定義する。本稿では配置操作側の関心に絞る。

### アンカー

配置操作と `patchComponent` はいずれも **Slot id と Component id の組み合わせ**で対象を指定する。Slot id で操作対象の Slot を、Component id でその Slot 内のコンポーネントを特定する。Component id は `insertBeforeComponent` / `insertAfterComponent` では挿入位置の基準（アンカー）として、`removeComponent` / `replaceComponent` / `patchComponent` では対象コンポーネントとして機能する。Slot id / Component id の命名規則は doc24 で定義する。

### 未登録 id への挙動

参照する Component id が指定 Slot 内で未登録の場合、Framework は呼び出し時に例外を投げる。対象が存在しない状態で続行すると、Customize の意図が未適用のまま画面が表示され、Core 更新での id 変更や typo を気付かず運用するリスクがある。`removeComponent` も同様に扱う（未登録 id への remove は no-op ではなく例外）。

## Core / Customize の役割分担

### Core: 標準構成を積み上げる

Slot を定義し、`addComponent` で標準構成を積み上げる。

```ts
registerComponents('product.basicInfo.fields')
  .addComponent('product.name', TextInputField, { label: '商品名', validation: z.string().max(100) })
  .addComponent('product.code', TextInputField, { label: '商品コード', validation: z.string().max(20) })
  .addComponent('product.description', TextareaField, { label: '説明' });
```

### Customize: 標準構成を改変する

5 操作すべて（`addComponent` / `insertBeforeComponent` / `insertAfterComponent` / `removeComponent` / `replaceComponent`）を使って、Core が積み上げた標準構成を改変する。

```ts
insertAfterComponent('product.basicInfo.fields', 'product.name', { id: 'product.subtitle', component: SubtitleField });
removeComponent('product.basicInfo.fields', 'product.sku');
replaceComponent('product.basicInfo.fields', 'product.description', CustomDescriptionField);
addComponent('product.basicInfo.fields', 'product.note', NoteField);
```

### 並び順の指定方針

既存要素に対する相対位置で挿入する場合は `insertBeforeComponent` / `insertAfterComponent` を用いる（既存の Component id をアンカーとして位置を明示する）。

複数モジュールから同一 Slot に `addComponent` で登録する場合は、各登録に `options.order` を明示することで絶対値で並び順を決める（詳細は doc24「複数回呼び出しの挙動と並び順」参照）。

## replaceComponent の位置づけ

`removeComponent + insertBeforeComponent/insertAfterComponent` の組み合わせでも同じ位置での置換は表現できるが、`replaceComponent` は独立した操作として提供する。

- **意図の明示性**: 「差し替え」という操作の意図が 1 行で伝わる
- **EC 実務の頻度**: 同じ位置で中身だけ差し替える要求が多い
- **可読性**: `remove + insert` の組み合わせより読みやすい

`replaceComponent` は位置を維持した 1 対 1 の差し替えを扱う。

## registry / collector 型の設計根拠

管理画面 UI の標準構成は **registry / collector 型** で持つ。Schema Builder 型は採らない。collector の入口は `Component` で統一する。

### registry / collector 型を採る理由

- React の JSX / component の柔軟性を殺しにくい
- DSL 設計を先に背負わずに済む
- EC 実務で多い小改修に向く
- 「枠をつくって、そこにパカパカはめ込む」という既存の発想と連続している

### Schema Builder 型を採らない理由

- field 種別の固定化（text / number / select 等を DSL に組み込むと拡張が重くなる）
- layout / visibility / validation の DSL 化で宣言量が爆発する
- state ownership との接続で DSL が state を管理し始めると責務が曖昧になる
- DSL で書けない UI が出たときに JSX 直書きへ逃げる道がない

### collector の入口は Component で統一する

collector の関数名は `field / block / page` で分けない。主語は `Component` に揃える。収集・配置の関数に粒度の違いを持ち込むと、同じ操作が階層ごとに別名になり Customize 開発者の認知負荷が増える。登録されるものの粒度（field / section / page）は JSX 側の構造に反映し、配置操作 API はフラットに保つ。
