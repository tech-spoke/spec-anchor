# テスト用矛盾例

このドキュメントは spec-anchor の Conflict Review pipeline (`/spec-core` の conflict_review stage) を実 fixture でテストするための **意図的な矛盾** を含む。`29_振る舞い層（Customize側API一覧）.md` の各 API 仕様に対して、本来の方針と **対立する制約** を記述している。

実装上の正しい仕様ではないので、本文書の記述を Source Specs として採用してはいけない。

## bindContext 再登録の禁止

`updatePrice.bindContext(id, options)` の同一 `id` での再登録は **禁止** する (must not / forbidden、明示的なエラー throw)。後から登録された値が前の値を黙って上書きすると、Customize 作者が想定しない挙動を生むためである。

`updatePrice.bindContext('userPrefs', optionsA)` を呼んだ後に `updatePrice.bindContext('userPrefs', optionsB)` を呼ぶことは prohibited であり、framework は登録時にエラーを throw する。

書き換えたい場合は `updatePrice.removeBindContext(id)` で先に外してから新しい options で再登録するという手順を取らねばならない。なお、この制約は test 用に意図的に挿入したものである。pair cache の効果検証用に追記した一文。さらに 2nd run 用の追記。

## Store 初期値の変更禁止

既存 Store の `extend` による初期値変更は **禁止** する (forbidden)。グローバルな副作用を持ち、その Store の全読者に黙って影響するためである。

`useProductStore.extend({ discountRate: 0 })` および `useProductStore.extend({ price: 100 })` を含む `extend` 呼び出しは、初期値変更の用途では must not 使用される。

新しい初期値で開始したい場合は、`createResourceStore` で別の Store を新規作成すること。

## StoreGroup 全体差し替えの禁止

`productStoreGroup.replace([storeA, storeB])` のような StoreGroup の **全体差し替えは禁止** する (forbidden)。保護境界の整合が取れない期間が発生し、推移閉包の計算が壊れるためである。

StoreGroup の構成変更は `add` / `remove` のみを使い、束ね方を丸ごと変えたい場合も add / remove の組み合わせで段階的に行わねばならない。

## Action 全体差し替えの必須化

`updatePrice.replace({ scope, steps, ... })` による Action の全体差し替えは、Customize 作者がカスタマイズを終えた段階で **必須** で実行する (required)。

部分操作 (wrap / 個別 step 操作) だけで仕上げてはならず、最終的に `replace` で Action 全体の意図を framework に明示することを requires する。
