# コンポーネント層（基本構造）

## 目的

管理画面UIの拡張可能な階層を、同じ再帰構造で構成する。

ページ・セクション・フィールドのどの階層でも、同じ「JSX レイアウト + Slot + registerComponents」のパターンを適用する。特別扱いの階層を作らないことで、Customize の拡張点を一貫した手段で提供する。

App.tsx（最上位・ルーティング）は Copy & Own 専用で、この再帰構造の対象外とする。

### スコープ

この文書で扱うもの:

- 全コンポーネントに共通する再帰構造
- Slot の役割と id
- Component の id 命名規則
- registerComponents の役割

この文書で扱わないもの:

- 配置の編集操作（insertBefore / insertAfter / remove / replace）— doc25
- 設定変更（patchComponent）— doc26
- Core 側の記述規約（CommonFieldProps / `useMain` / `behaviors` など）— doc26
- ディレクトリ物理配置（doc22）
- ルーティング機構（標準の React Router をそのまま用いる。App.tsx レベルは Copy & Own のみで、Slot 機構の対象外）

---

## 基本構造

全コンポーネントは次の再帰パターンで構成される。

```text
[コンポーネントの JSX レイアウト]
  └ [Slot]                              拡張可能な領域
      └ [登録コンポーネント群]          registerComponents で配置される
          └ [コンポーネントの JSX レイアウト]
              └ [Slot]
                  └ [登録コンポーネント群]
                      └ ...
```

- コンポーネントは JSX でレイアウトを定義する
- レイアウト内に拡張点として `Slot` を配置する
- 各 Slot には `registerComponents` で 0 個以上のコンポーネントが登録される
- 登録されたコンポーネントは、再度この構造を持つ

ページ・セクション・フィールドのどの階層でも、この構造は同じ。App.tsx はこの再帰の起点ではなく、標準 Router で各ページをマウントする薄い層として独立している（Copy & Own のみ）。

---

## 階層ごとの例

### トップレベル（App）

App.tsx は標準の React Router をそのまま用いてページを配置する。Slot 機構の対象外。

```tsx
// app/App.tsx
import { BrowserRouter as Router, Route } from 'react-router-dom';
import { AppShell } from '@core/components/shell/AppShell';
import { ProductEditPage } from '@core/components/products/pages/ProductEditPage';
import { ProductListPage } from '@core/components/products/pages/ProductListPage';

export function App() {
  return (
    <AppShell>
      <Router>
        <Route path="/products/:id/edit" component={ProductEditPage} />
        <Route path="/products"          component={ProductListPage} />
      </Router>
    </AppShell>
  );
}
```

App.tsx の変更（ルート追加・差し替え・レイアウト変更）は Copy & Own で行う。Slot による登録・差し替えの対象とはしない。

本稿で扱う再帰構造は App.tsx の一段下、各ページから始まる。

### ページ

ページはトップレベル Slot に登録されるコンポーネント。自身の JSX 内に複数の Slot を配置してもよい。

```tsx
// core/components/products/pages/ProductEditPage.tsx
import { Slot, registerComponents } from '@core/infra/components';
import { saveProduct } from '@core/actions/saveProduct';
import { BasicInfoSection } from './sections/BasicInfoSection';
import { MetadataSection } from './sections/MetadataSection';
import { PriceSection } from './sections/PriceSection';
import { StockSection } from './sections/StockSection';

export function ProductEditPage(props) {
  const useMain = props.useMain ?? useProductEditMain;
  const ui = useMain(props);

  return (
    <PageLayout>
      <Tabs activeTab={ui.activeTab} onChange={ui.setActiveTab}>
        <Tab id="basic">
          <Slot id="product.edit.basic" />
        </Tab>
        <Tab id="pricing">
          <Slot id="product.edit.pricing" />
        </Tab>
      </Tabs>
      <Footer>
        <Button onClick={() => saveProduct()}>保存</Button>
      </Footer>
    </PageLayout>
  );
}

registerComponents('product.edit.basic')
  .addComponent('product.basicInfo', BasicInfoSection)
  .addComponent('product.metadata', MetadataSection);

registerComponents('product.edit.pricing')
  .addComponent('product.price', PriceSection)
  .addComponent('product.stock', StockSection);
```

### セクション

セクションはページの Slot に登録されるコンポーネント。自身の JSX 内に Slot を配置してフィールドを受ける。

```tsx
// core/components/products/pages/sections/BasicInfoSection.tsx
import { z } from 'zod';
import { Slot, registerComponents } from '@core/infra/components';
import { TextInputField } from '@core/components/shared/fields/TextInputField';
import { TextareaField } from '@core/components/shared/fields/TextareaField';

export function BasicInfoSection() {
  return (
    <Card>
      <CardHeader>基本情報</CardHeader>
      <CardBody>
        <Slot id="product.basicInfo.fields" />
      </CardBody>
    </Card>
  );
}

registerComponents('product.basicInfo.fields')
  .addComponent('product.name', TextInputField, { label: '商品名', validation: z.string().max(100) })
  .addComponent('product.code', TextInputField, { label: '商品コード', validation: z.string().max(20) })
  .addComponent('product.description', TextareaField, { label: '説明' });
```

### フィールド

フィールドは `shared/fields/` に置かれる汎用パーツ、または feature 固有のフィールド。どちらもセクションの Slot に登録される。

拡張余地が要らなければ Slot を持たない leaf として記述する。拡張余地を残したい場合は Slot を配置できる。

```tsx
// core/components/shared/fields/TextInputField.tsx
import { Slot } from '@core/infra/components';

export function TextInputField(props) {
  const useMain = props.useMain ?? useTextInputFieldMain;
  const ui = useMain(props);

  return (
    <FieldWrapper label={props.label} helpText={props.helpText}>
      <Slot id={`${props.id}.prefix`} />
      <input value={props.value} onChange={props.onChange} />
      <Slot id={`${props.id}.suffix`} />
    </FieldWrapper>
  );
}
```

## Slot

### 役割

拡張可能な領域を JSX 上で宣言する。その領域に登録されたコンポーネント群を順に描画する。

### 属性

| 属性 | 型 | 用途 |
|---|---|---|
| `id` | `string` | この Slot を識別する（UI 構造由来の命名。後述） |

Slot は JSX 上で子要素を持たない。描画する内容は `registerComponents(slotId)` で登録されたコンポーネント群に一意に決まる。

---

## id の設計

### Slot の id

UI 構造由来の命名。画面内の位置を表現する。

例: `app.pages`、`product.edit.basic`、`product.basicInfo.fields`

Slot は「どの場所か」を指すものであるため、UI 構造を命名に含めることが妥当である。

### Component の id

ドメインモデル由来のフラットな命名。

形式: `{resource}.{要素}`

例: `product.name`（フィールド）、`product.basicInfo`（セクション）、`product.edit`（ページ）

#### 根拠

- **推測しやすさ**: ドメイン由来の命名は Customize 開発者が id を推測できる
- **安定性**: セクションの分割・統合や画面構成の変更で id が変わらない。UI 構造由来（`product-edit.basicInfo.name` 等）だと、セクション再構成のたびに Customize が壊れる
- **衝突回避**: 配置操作 API と `patchComponent` は対象を Slot id と Component id の両方で指定する（doc25）。同一 Slot 内で Component id が一意であれば衝突しない

### id 空間の分離

Slot id と Component id は別の名前空間。形式上は似ていても目的が異なる。

| id | 意味 | 命名由来 |
|---|---|---|
| Slot id | どの場所か | UI 構造 |
| Component id | 何のコンポーネントか | ドメインモデル |

## registerComponents

### シグネチャ

```ts
registerComponents(slotId: string)
  .addComponent(componentId: string, Component, props?, options?)
  .addComponent(componentId: string, Component, props?, options?)
  ...
```

| 引数 | 型 | 用途 |
|---|---|---|
| `slotId` | `string` | 登録先の Slot id |
| `componentId` | `string` | コンポーネントの id（Customize がアンカーとして使う） |
| `Component` | React コンポーネント | 登録するコンポーネント |
| `props` | `object?` | コンポーネントの初期 props（省略可） |
| `options` | `{ order?: number }?` | 登録メタデータ。`order` は並び順の明示指定（省略可） |

### 戻り値

`registerComponents(slotId)` は builder を返し、`.addComponent(...)` は同じ builder を返す。chaining 専用で、外部に有用な公開メンバを持たない（型上は `{ addComponent: (...) => this }` 相当）。

途中参照（変数に束ねて後から `.addComponent` を呼ぶ記述）や、登録結果を照会する API は提供しない。Core 側は決定的に標準構成を積み上げ、Customize 側は配置操作 API（doc25）で介入するため、どちらの経路でも builder を再参照する用途がない。照会 API は消費者が特定された時点で追加検討する。

### 登録整合性

同一 Slot 内で componentId が重複した場合、Framework が登録時に例外を投げる。登録結果の確認手段ではなく、登録 API 自身の整合性保証として位置づける。異なる Slot に同じ componentId が登録されることは許容する（配置操作 API は Slot id と Component id の両方で対象を指定するため、Slot を跨いで衝突しない）。

### 評価タイミング

`registerComponents` の呼び出しは、コンポーネントを定義するモジュールの評価時に実行される副作用。
モジュールを import すれば登録が完了する。

### 複数回呼び出しの挙動と並び順

同一 slotId に対する `registerComponents` の複数回呼び出しは append として動作する。先に評価された呼び出しの登録に、後から評価された呼び出しの登録が末尾へ追加される。ES Modules の評価順は仕様で決定的であるため、結果の並びは一意に決まる。

並び順は `options.order` で明示指定できる。

- chain 登録（`registerComponents().addComponent().addComponent()...`）の各 `addComponent` には暗黙の昇順 `order`（0, 1, 2, ...）が振られる
- `options.order` を明示した登録は、その数値で並ぶ
- 同一 `order` を持つ登録が複数ある場合は評価順に従う

同一 Slot に複数モジュールから登録する場合は、各登録で `options.order` を明示することを推奨する。明示しない場合は評価順で末尾に追加され、モジュールの import 順に依存した並びになる。

Customize が既存要素に対する相対位置で挿入する場合は `insertBeforeComponent` / `insertAfterComponent` を用いる（doc25）。`options.order` は複数モジュールから絶対値で並び順を指定する用途で、`insertBeforeComponent` / `insertAfterComponent` と使い分ける。

## 構造上の性質

### 全階層が同じパターン

ページ / セクション / フィールドのどの階層も「JSX レイアウト + Slot + registerComponents」で構成される。特別扱いの階層はない。

### 拡張点が JSX 上で可視

`<Slot id="...">` を見れば、そこが Customize 可能な拡張点だと即座にわかる。

### App レベルは再帰構造の外

App.tsx（Router + Route 配置）は Copy & Own でのみ変更する。再帰構造（Slot による拡張）は App の一段下、各ページから始まる。

App.tsx の Customize 需要は稀（レイアウト・ルート追加はほぼ発生しない）で、かつ App.tsx は薄い層であるため Copy & Own でも追随コストは小さい。代償として Core 側で新ページが追加されても shadow 済み App.tsx には自動反映されないが、発生頻度の低さから許容する。

## 未確定事項

| 項目 | 備考 |
|---|---|
| AppShell / ナビゲーション層の customize | App.tsx は Copy & Own 専用（再帰構造の対象外）として確定しているが、その直下の AppShell（ヘッダー・サイドバー・グローバルメニュー）を Slot 機構の対象とするかは未決。メニュー項目の追加・削除・並び替えは EC 実務で頻出する customize であり、Copy & Own のみに委ねると追随コストが増える。候補は (a) AppShell 内に Slot を置いて `addComponent` / `insertBeforeComponent` 等で扱う、(b) メニュー項目を preset 的な配列（`coreSidebarMenuFor()` 等）として export し `patchComponent` で上書きさせる、の 2 つ。いずれも Core 側 AppShell の設計判断を含むため、本稿で扱う再帰構造とは別に決める |
| Slot レイアウト機構 | 現状の Slot は「登録順に描画」のみで、grid / flex / 幅配分（カラム数・gap・grid span 等）は Slot を含む親 JSX 側で固定される。customize 側が「特定フィールドだけ幅 2 カラム」「レスポンシブでメディアクエリごとに並び方を変える」等を指定する手段は `CommonFieldProps.className` 経由の grid span 指定だけで、Slot 全体のレイアウトを差し替える正規の経路が無い。候補は (a) Slot に `layout` prop（列数・gap 等）を持たせて `patchComponent` で Slot 設定を上書き可能にする、(b) レイアウトは親 JSX で固定し customize は行わせない（ファイル単位 shadow のみ許容）、(c) Slot の子（登録コンポーネント）の className 経由のみで済ませる、の 3 つ |
