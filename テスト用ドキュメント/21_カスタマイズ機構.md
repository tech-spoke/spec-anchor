# カスタマイズ機構

## 目的

doc20 が定めたカスタマイズの構造原則（ミラー配置 + サフィックス規約）を実現する仕組みを定める。

### スコープ

この文書で扱うもの:

- shadow（copy & own）の解決方法
- patch（差分適用）の解決方法
- グローバル→ローカルの適用順序の保証方法

この文書で扱わないもの:

- カスタマイズの原則・思想（doc20）
- patch で使う API の詳細（doc28 / doc29 / doc30）
- ディレクトリの物理配置（doc22）

### 前提

doc20 のカスタマイズ構造原則を前提とする:

- customize 側のコードは `customize/` に、core のディレクトリ構造をミラーした配置を取る
- `.patch.ts` — core を import し、API で差分を当てる（patch）
- `.ts` / `.tsx`（サフィックスなし） — core の同名ファイルを丸ごと上書きする（shadow）

---

## shadow の解決方法

shadow は、バンドラーのパスエイリアス解決で実現する。
`@core/*` の解決先を `customize/*` → `core/*` の順にすることで、
customize 側にファイルがあればそちらが優先される。

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@core/*": [
        "customize/*",
        "core/*"
      ]
    }
  }
}
```

- `customize/actions/sendApi.ts` が存在すれば、`import ... from '@core/actions/sendApi'` はそちらを解決する
- 存在しなければ `core/actions/sendApi.ts` にフォールバックする
- core 側のファイルは一切変更しない。shadow はバンドラーレベルの解決であり、git 上の core は upstream のまま保たれる

### shadow ファイルの制約

- shadow ファイルは core の同名ファイルと同じ export インターフェースを維持する責務を持つ。呼び出し元が期待する import が壊れないようにするのは Customize 作者の責務である
- shadow したファイルは core の更新に自動追従しない。upstream の変更を取り込むかどうかは Customize 作者が判断する

## patch の解決方法

patch ファイル（`.patch.ts`）は side-effect import として読み込まれる。
core のモジュールを import し、API（doc29 / doc30）で差分を当てる。

```ts
// customize/actions/updatePrice.patch.ts
import { updatePrice } from '@core/actions/updatePrice';

updatePrice.logic(async (ctx, next) => {
  // カスタマイズされたロジック
  await next();
});
```

```ts
// customize/store/useProductStore.patch.ts
import { useProductStore } from '@core/store/useProductStore';

declare module '@core/store/useProductStore' {
  interface ProductStoreState {
    discountRate: number;
  }
}

useProductStore.extend({ discountRate: 0 });
```

### patch ファイルの発見方法

策定中。以下のいずれかの方式を検討する:

- **auto-scan**: ビルド時に `customize/` 配下の `*.patch.ts` を自動収集し、エントリポイントに import 文を生成する
- **手動 import**: `app/main.ts` に Customize 開発者が明示的に import 文を記述する

### shadow と patch の共存

同一の対象に対して shadow ファイルと patch ファイルが同時に存在する場合の扱いは策定中。

## 適用順序の保証

Customize の変更にはグローバル変更とローカル変更があり、グローバル変更を先に適用する必要がある（根拠は doc22 参照）。

### 3フェーズの実行順序

Customize patch の適用は以下の3フェーズで行われる。

| フェーズ | 内容 | 何が起きるか |
|---|---|---|
| 1. Core 定義 | Store / StoreGroup / Preset のモジュールが ES module 評価で生成される | `createResourceStore`, `defineStoreGroup`, `definePreset` の呼び出しにより、操作対象のオブジェクトが存在するようになる |
| 2. グローバル変更 | `@core/globals/` の import で Customize のグローバル patch が実行される | Store 拡張（`.extend`）、StoreGroup 変更（`.add` / `.remove` / `.replace`）、Preset 変更（`.prepend` / `.append` 等）、Action の replace / disable |
| 3. Action 登録 + ローカル変更 | `@core/actions/` の import で Core が Action を登録し、Customize のローカル patch が適用される | `actionRegistry.register()` による Action 生成、個別 Action への step / scope 操作 |

フェーズ 1 は ES module の import 解決で自然に起きる（globals や actions のファイルが Store / Preset モジュールを import するため、先に評価される）。
フェーズ 2→3 の順序は core 側のエントリポイントの import 順序で保証する。

### 保証方法

core は `globals/` ディレクトリを持つ（core 側では中身は空）。
core のエントリポイントが `globals/` → `actions/` の順で import する。

```ts
// core/setup.ts
import '@core/globals/...';    // フェーズ 2: グローバル変更（core では空、customize が shadow / patch で注入）
import '@core/actions/...';    // フェーズ 3: Action 登録 + ローカル変更
```

`@core/globals/...` はフォールバック解決により `customize/globals/` のファイルが優先される。
customize 側がグローバル patch を `customize/globals/` に置けば、core のエントリポイントの import 順序によって、Action 登録およびローカル変更より先に適用される。

この順序が守られない場合に発生する問題の詳細は doc22 を参照。
