# ディレクトリ構造

## 目的

管理画面のディレクトリ構造のうち、確定している部分を定める。
確定していない領域は末尾に明記する。

### スコープ

この文書で扱うもの:

- ルートレベルのディレクトリ境界（core / customize / framework）
- パスエイリアス
- Core / Framework / Customize のファイル配置
- グローバル変更とローカル変更の適用順序制約の根拠

この文書で扱わないもの:

- カスタマイズの構造原則（doc20）
- patch / shadow の解決方法、適用順序の保証方法（doc21）
- backend のディレクトリ構造
- storefront のディレクトリ構造

---

## ルートレベル構造

```text
app/              … アプリケーションエントリポイント
core/             … upstream / git フォーク管理対象
customize/        … 利用者カスタマイズ領域
framework/        … Framework 内部実装
```

- `core/` は標準構成を提供する。直接編集してカスタマイズしない（doc20）
- `customize/` は core の標準構成に差分を当てる領域（doc20）
- `framework/` は doc28 / doc30 が定義した API の内部実装。Core 開発者も Customize 開発者も直接触らない（doc31）
- `app/` はモジュール読み込み順序を制御するエントリポイント（doc30）

### パスエイリアス

| エイリアス | 解決先 | 用途 |
|---|---|---|
| `@core/` | `customize/*` → `core/*`（オーバーライド） | Core の公開 API・Store・Action・Component。customize 側に shadow ファイルがあればそちらを優先する（doc21） |
| `@customize/` | `customize/` | Customize 独自のコード（配置先は策定中） |
| `@framework/` | `framework/` | Framework 内部実装（doc31 参照） |

## Core 内部構造

doc28 のコードパスから確定する構造。

```text
core/
  infra/                        … Framework が提供する公開 API 型・関数
    resourceStore.ts            … createResourceStore, ResourceStore, ResourceStoreMeta
    storeGroup.ts               … defineStoreGroup, StoreGroup
    actionRegistry.ts           … actionRegistry.register, ActionProxy, StepDefinition,
                                  StepMiddleware, ActionResult, StepCollection
    actionScope.ts              … ActionScope, StoreRef
    actionContext.ts            … ActionContext, StoreHandle
    errors.ts                   … ValidationError, BusinessError
    hooks.ts                    … useActionStatus
  globals/                      … Customize のグローバル変更フック（Core では空）
  store/                        … Store 定義
  middlewares/                  … 再利用可能な step 部品（middleware factory）
  presets/                      … step 列パターン（definePreset でラップ）
  actions/                      … Action 定義（actionRegistry.register + 名前付き export）
  components/                   … React Component（feature-based 構造。後述）
  errors/                       … BusinessError の派生クラス
```

### 各ディレクトリの責務

| ディレクトリ | 何を置くか | 誰が書くか |
|---|---|---|
| `infra/` | Framework が提供する型・関数の定義 | Framework |
| `globals/` | Customize のグローバル変更フック。Core では空。エントリポイントの import 順序で適用順序を保証する（doc21） | Core |
| `store/` | `createResourceStore` で作る資源 Store。`defineStoreGroup` で作る StoreGroup。UI 状態 Store（素の `create`） | Core |
| `middlewares/` | `zodValidation` のような再利用可能な step factory | Core |
| `presets/` | middleware + logic の並びを `definePreset` でパターン化したもの | Core |
| `actions/` | `actionRegistry.register` で登録する Action | Core |
| `components/` | Store を読み Action を呼ぶ React Component | Core |
| `errors/` | `BusinessError` の派生クラス（ドメイン固有エラー） | Core |

`infra/` の中身は Framework が提供するもので、Core 開発者は import して使う側である。
`infra/errors.ts` の `ValidationError` / `BusinessError` は基底クラスであり、
Core が定義するドメイン固有エラーは `errors/` に置く。

### components/ の内部構造

`components/` は feature-based + コロケーションで構成する。

```text
core/components/
  products/                        … feature（商品）
    pages/
      ProductEditPage.tsx
      sections/
        BasicInfoSection.tsx
        PriceSection.tsx
        StockSection.tsx
      ProductListPage.tsx
      sections/
        SearchPanel.tsx
    shared/                        … feature 内で複数ページが共有するコンポーネント
  orders/                          … feature（受注）
    pages/
      OrderEditPage.tsx
      sections/
        ShippingSection.tsx
      OrderListPage.tsx
      sections/
        SearchPanel.tsx
  shared/                          … feature 横断の共有コンポーネント
    fields/
      TextInputField.tsx
      NumberInputField.tsx
      SelectField.tsx
      DatePickerField.tsx
```

#### 構成原則

| 原則 | 内容 |
|---|---|
| feature 単位 | resource（products, orders 等）ごとにディレクトリを分ける |
| コロケーション | page とその section を近くに置く。section は page の下にネストする |
| feature 間 import 禁止 | `products/` → `orders/` の直接 import は禁止。依存の方向は常に `feature → shared` |
| 共有の段階 | feature 内の複数ページで共有 → `{feature}/shared/`。feature をまたぐ → `components/shared/` |

共有フィールドコンポーネント（`TextInputField` 等）は `shared/fields/` に置く。addComponent の第3引数（props）により、共有コンポーネントを設定違いで使い回す（addComponent の API は doc24、props の型定義は doc26 を参照）。

## Customize 内部構造

doc20 のカスタマイズ構造原則に基づき、customize 側は core のディレクトリ構造をミラーした配置を取る。
`customize/` 自体が core のミラーであり、patch / shadow / 独自ファイルのいずれも同じディレクトリに置く。

```text
customize/
  globals/                       … グローバル変更の patch を集約（Store 拡張、Preset 変更等）
  store/                         … Store の拡張（patch）/ 差し替え（shadow）/ 独自 Store の新規作成
  middlewares/                   … middleware の追加・差し替え
  presets/                       … preset の変更（patch）または差し替え（shadow）
  actions/                       … Action の操作（patch）/ 差し替え（shadow）/ 独自 Action の新規作成
  components/                    … Component の設定変更（patch）/ 差し替え（shadow）/ 独自 Component。core と同じ feature-based 構造をミラーする
  errors/                        … エラークラスの追加・差し替え
```

ファイルのサフィックスで patch（`.patch.ts`）か shadow（`.ts` / `.tsx`）かを宣言する（doc20 / doc21）。
独自ファイル（core に対応するファイルがない新規作成）もサフィックスなしの `.ts` / `.tsx` で、同じディレクトリに置く。

`globals/` は core に対応するディレクトリを持たないが、グローバル変更（Store 拡張、StoreGroup 変更、Preset 変更等）をローカル変更（個別 Action の step 操作）と分離して集約するためのディレクトリである。

### グローバル変更とローカル変更の区分

Customize の振る舞い変更はグローバル変更とローカル変更に分かれる。
この区分はディレクトリ構造ではなく、操作の影響範囲によって定まる。

| 種類 | 影響範囲 | 操作例 |
|---|---|---|
| グローバル | 複数 Action / 全読者に波及 | 下表 |
| ローカル | 対象 Action のインスタンスだけ | `updatePrice.logic(fn)` / `updatePrice.scope.internalStores.add(store)` |

#### グローバル変更に該当する操作

| 対象 | 操作 | 波及範囲 |
|---|---|---|
| Store | `useProductStore.extend({ discountRate: 0 })` | その Store の全読者 |
| StoreGroup | `productStoreGroup.add(store)` / `.remove()` / `.replace()` | Group を参照する全 Action |
| Preset | `fieldInputPreset.prepend(step)` / `.remove()` / `.replace()` 等 | Preset を使う全 Action |
| Action replace / disable | `updatePrice.replace({ ... })` / `updatePrice.disable()` | 全呼び出し元 |

これらはすべて「1つの操作で複数の Action や読者の振る舞いが変わる」操作である。
ローカル変更と混在すると、適用順序に依存した不整合が発生する（次節参照）。

## 適用順序制約

Customize の patch は以下の順序で適用されなければならない。
順序の保証方法は doc21 で定める。

### 順序制約

Customize patch の適用は3フェーズで行われる（詳細と保証方法は doc21 参照）。

| フェーズ | 内容 |
|---|---|
| 1. Core 定義 | Store / StoreGroup / Preset のモジュールが ES module 評価で生成される |
| 2. グローバル変更 | Customize のグローバル patch が実行される（Store 拡張、StoreGroup 変更、Preset 変更、Action replace / disable） |
| 3. Action 登録 + ローカル変更 | Core が Action を登録し、Customize のローカル patch が適用される |

### グローバル → ローカルの根拠

グローバル変更は複数 Action の基盤（Store shape、StoreGroup 構成、preset step 列）を変える。
ローカル変更はそれらの基盤が確定した状態で個別 Action を操作する。

グローバル変更を先に適用しないと、以下の不整合が発生する:

1. Customize patch X が `updatePrice.logic(fn)` を実行（ローカル）
2. Customize patch Y が `fieldInputPreset.prepend({ id: 'audit', fn })` を実行（グローバル）

この順序では、patch X の時点で `updatePrice` の steps に `audit` は存在しない。
patch Y のグローバル変更は、Y の後に評価される Action には効くが、
X が既に操作した `updatePrice` には効果が不完全になる。

同様に:

- `useProductStore.extend({ discountRate: 0 })` より先に `updatePrice.logic(fn)` が走ると、logic 内で `discountRate` にアクセスできない
- `productStoreGroup.add(useMemberStore)` より先に `updatePrice.scope.internalStores.add(...)` が走ると、scope の前提が異なる

全グローバル変更が適用された状態でローカル操作が走ることを保証する必要がある。
