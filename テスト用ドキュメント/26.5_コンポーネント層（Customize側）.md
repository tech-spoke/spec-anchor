# コンポーネント層（Customize 側）

## 目的

Customize 開発者が component 層の customize 手段を選び、使い分けるための入口文書。軽い→重いの進行、16 分類ごとの契約相性、典型シナリオのマッピングを集約する。

### スコープ

この文書で扱うもの:

- component 層 customize の進行（軽い→重い）と手段の一覧
- 16 分類ごとの customize 契約相性
- 典型シナリオごとの手段選択と最小コード例

この文書で扱わないもの:

- 配置操作 API の仕様（`addComponent` / `insertBeforeComponent` / `insertAfterComponent` / `removeComponent` / `replaceComponent`）— doc25
- Core 側の記述規約（PROPS / 処理 / 合成 hook / 部品 hook（状態・ブリッジ・instance 生成・進捗 等））と介入経路の原理 — doc26
- customize の上位俯瞰（component + 振る舞いを横断する段階）— doc20
- 振る舞い層の customize（Action id 直接アクセス、step 操作）— doc30

## 1. customize の進行

### 1.1 手段の一覧

| 粒度 | 手段 | 介入対象 |
|---|---|---|
| props 単位 | `patchComponent('slotId', 'componentId', { label, validation, ... })` | 登録 props の部分上書き（doc26 §3.1）。merge は top-level で単純置換（doc31）|
| 部品 hook 単位 | `patchComponent('slotId', 'componentId', { behaviors: { useState: myState, useValue: myValue } })` | `behaviors` 配下の差し替え（doc26 §4.1）。merge は `behaviors` 配下のキーで単純置換（doc31）。元を活かす拡張は §3.5 参照 |
| 合成 hook 全体 | `patchComponent('slotId', 'componentId', { useMain: myMain })` | 合成 hook 全体の差し替え（doc26 §4.1）。merge は top-level で単純置換（doc31）|
| 配置変更 | `insertBeforeComponent` / `insertAfterComponent` / `removeComponent` / `replaceComponent`（doc25）| Slot 内の並びの操作 |
| コンポーネント単位 | `replaceComponent('slotId', 'componentId', MyComp)` | コンポーネント丸ごと差し替え |
| Store 書き込み挙動 | `updatePrice.logic(fn)` / `.wrap(fn)` 等 | Action パイプラインのミドルウェア（doc30）|
| ファイル単位 | Copy & Own（shadow）| ファイル丸ごと差し替え |

配置変更は doc25、Store 書き込み挙動は doc30 の API を借用する。component 層に固有の介入手段は `patchComponent` と `replaceComponent`。介入が「なぜ通るか」の原理は doc26 §6 を参照。

### 1.2 軽い→重いの進行

```
軽い                                                                           重い
  ↓                                                                             ↓
patchComponent(props) → behaviors 差し替え → useMain 差し替え →
  配置変更（insertBefore / insertAfter / remove）→ Action ミドルウェア →
  replaceComponent → Copy & Own
```

大半の customize は `patchComponent`（props + 部品 hook）と配置変更で収まる。`replaceComponent` と Copy & Own は最終手段。

doc20 「カスタマイズの段階」の component 層特化版。component と振る舞いを横断する上位俯瞰は doc20 を参照。

## 2. 分類ごとの customize 契約相性

customize 側からの介入手段（patchComponent の props 上書き・合成 hook 差し替え・replaceComponent）との相性を 16 分類ごとに示す。分類番号は doc23 §振る舞い分類に対応。

### 2.1 props 上書きで完結する

1 / 2 / 3 / 6 / 10 / 12 / 13 / 14。`label` / `placeholder` / `options` / `format` 等の宣言的設定が中心で、`patchComponent('slotId', 'componentId', { ... })` による部分上書きで網羅できる。

### 2.2 config-like props で完結する

- 9 テーブル: `columns` 配列を props として受け取る。`patchComponent('slotId', 'componentId', { columns: [...coreInlineTableColumnsFor(slotId), myColumn] })` のように Core 既定を関数で取得して spread / filter で上書きする（配列系 props の customize 契約は doc26 §3.6）
- 11 チャート: `data` / 軸 / 色の props を差し替える

### 2.3 ライブラリ差し替えの介入経路を持つ

外部ライブラリを使う分類 4・5 は、Core 側が部品を複数段に分けて提供している（doc26 §4.6）。customize 側は部品単位で差し替えられる。

- 4 リッチテキスト（Tiptap）: Core が `useTiptapEditor`（instance 生成）と `useRichTextFieldBridge`（RHF 接続）を提供する。`behaviors.useEditor` で Tiptap → Lexical 等のライブラリ差し替え、`behaviors.useBridge` で RHF 接続ロジックの差し替えができる。`extensions` は `coreRichTextExtensionsFor()` で Core 既定を取得し spread / filter で上書きする（配列系 props の customize 契約は doc26 §3.6）
- 5 メディアアップロード: Core が `useUppyInstance`（Uppy 生成、3 コンポーネント共有）、`use{Xxx}FieldBridge`（complete event 結線）、`useUploadProgressState`（進捗 state、3 コンポーネント共有）を提供する。`behaviors.useUploader` で Uppy → 独自 S3 クライアント等の差し替え、`behaviors.useBridge` で URL 反映ロジックの差し替え、`behaviors.useProgress` で進捗 state の形の差し替えができる。アップロード本体は Uppy が直接サーバー送信し、complete event でブリッジが `field.onChange` を呼ぶ。Action パイプラインは保存ボタン押下時の Store 反映を担当する（doc28 参照）

### 2.4 replaceComponent が自然な粒度になる

- 7 リピーター: 行テンプレート自体のカスタマイズは `patchComponent('{slotId}', 'items', { row: CustomRow })` で行を差し替える。行が独立 component として `RepeaterRowProps` を受け取る契約で、部分差し替えの粒度が一致する（doc26 §3.7 リピーター）
- 16 ツリー操作: カスタマイズ点（collision 戦略、水平ドラッグの深さ変更判定、親子一括移動）がハンドラ主体で宣言的 props に馴染まないため、`patchComponent` の props 上書きでは意味ある介入ができない。`replaceComponent` でブロック差し替え、または Copy & Own（shadow）で対応する（doc20 「カスタマイズの段階」参照）

## 3. 典型シナリオ

### 3.1 label / placeholder の変更

```ts
patchComponent('product.basicInfo.fields', 'product.name', {
  label: '商品タイトル',
  placeholder: '例: プレミアムTシャツ',
});
```

### 3.2 validation の追加・変更

```ts
patchComponent('product.basicInfo.fields', 'product.code', {
  validation: z.string().regex(/^PRD-\d{6}$/),
});
```

### 3.3 セクションへのフィールド追加

```ts
insertAfterComponent(
  'product.basicInfo.fields',
  'product.name',
  { id: 'product.subtitle', component: TextInputField, props: { label: 'サブタイトル' } }
);
```

### 3.4 ライブラリ instance の差し替え（分類 4・5）

```ts
// Tiptap を Lexical に差し替え
patchComponent('product.description', 'product.description', {
  behaviors: { useEditor: useLexicalEditor },
});

// Uppy を独自 S3 クライアントに差し替え
patchComponent('product.media', 'product.images', {
  behaviors: { useUploader: useCustomS3Uploader },
});
```

ブリッジ部品（`useBridge`）・進捗部品（`useProgress`）は Core のまま再利用される。差し替え対象は instance 生成部品だけ（doc26 §4.6.1 の customize キー一覧参照）。

### 3.5 元の部品 hook を差し替え・拡張する

部品 hook の挙動に項目を足す・前後に処理を挟むといった「元を活かす拡張」は、Core の named export を import してラップした hook を `behaviors` に渡す。

```ts
import { useState } from 'react';
import { useTextInputFieldState } from '@core/components/shared/fields/TextInputField';

function useExtendedState() {
  const base = useTextInputFieldState();
  const [logged, setLogged] = useState(false);
  return { ...base, logged, log: () => setLogged(true) };
}

patchComponent('product.basicInfo.fields', 'product.name', {
  behaviors: { useState: useExtendedState },
});
```

plain 関数の場合も同じパターン:

```ts
import { formatPrice } from '@core/components/shared/fields/PriceField';

const myFormat = (v: number) => `¥${formatPrice(v)}（税込）`;

patchComponent('product.basicInfo.fields', 'product.price', {
  behaviors: { format: myFormat },
});
```

Framework の merge 規則は単純置換（doc31「patchComponent の merge 規則」）。配列系 props の §3.6（doc26）の `coreXxxFor()` + spread パターンと同じ構図で、「追加」「拡張」は customize 側が import + ラップで構築する。

### 3.6 合成 hook（useMain）の差し替え

```ts
patchComponent('product.basicInfo.fields', 'product.price', {
  useMain: useCustomPriceFieldMain,
});
```

合成 hook を丸ごと差し替える。Core 側の部品 hook・plain 関数は `useCustomPriceFieldMain` 内で自由に呼び出せる（詳細は doc26 §4.1）。

### 3.7 リピーター行テンプレートの差し替え

```ts
patchComponent('order.items', 'order.items.repeater', {
  row: CustomOrderItemRow,
});
```

行 component は `RepeaterRowProps` を受け取る契約（doc26 §3.7 リピーター）。

### 3.8 コンポーネント丸ごと差し替え

```ts
replaceComponent('product.basicInfo.fields', 'product.description', CustomDescriptionField);
```

位置は維持したまま中身だけ差し替わる（doc25 §`replaceComponent` の位置づけ）。

### 3.9 ファイル単位の copy & own

customize 側ディレクトリに core と同名のファイルを配置する。ファイル全体が customize 側のものになる（doc20 「カスタマイズ構造原則」・doc21）。

```
customize/components/products/pages/sections/BasicInfoSection.tsx
```
