# フレームワーク実装仕様書

## 位置づけ

この文書は、Core 開発者の書き方（doc26 / doc28）と
Customize 開発者の書き方（doc26.5 / doc30）が外形のみを定めた
Framework 内部の挙動を実装仕様として確定する。

対象領域:

- **振る舞い層**: doc28 / doc30 が定義する Action / Store / Preset / StoreGroup 等の Framework 実装
- **コンポーネント層**: doc26 / doc26.5 が参照する `patchComponent` の merge 規則

いずれの層も「Framework 内部の実装は扱わない」としている箇所を、本稿が具体化する。

### 本稿で定義するもの

- **StepCollection** — step 列の共通データ構造
- **createResourceStore** の拡張（`.extend` 対応）
- **defineStoreGroup** の拡張（`add` / `remove` / `replace` 対応 + 循環検出）
- **flattenRefs** — StoreGroup の再帰展開
- **definePreset** — preset を操作可能オブジェクトにする仕組み
- **ActionProxy** — Action の Proxy 骨格と全操作
- **BindingRegistry** — bindContext の管理と解決
- **RuntimeMap** — ステップ間一時データの Map ラッパー
- **actionRegistry** — Action の登録と管理
- **パイプライン実行** — バッファモデルによる遅延 commit と compose
- **排他制御** — 排他 scope の重なりによる悲観ロック
- **useActionStatus** — Action の実行状態を購読する React hook
- **ctx.call と savepoint** — `ctx.call` による savepoint と失敗時の復元
- **シグナルディスパッチ** — `ctx.emit` → `on` 購読の伝播メカニズム
- **定義の確定と遅延適用** — freeze 不導入の判断と構造的安全性
- **patchComponent の merge 規則** — 登録 props と patch の 2 バケット単純置換 merge、および合成パターンの位置づけ

### 前提原則

本稿の設計は以下の原則に従う。

| # | 原則 | 出典 |

## StepCollection

StepCollection は、
`{ id, fn }` 形式の step 列を管理する
順序付きミュータブルコレクションである。

PresetTemplate と ActionProxy の steps が共有する基盤データ構造である。

### 内部構造

```ts
// framework/stepCollection.ts
import { StepDefinition, StepMiddleware } from '@core/infra/actionRegistry';

class StepCollection<TValue> {
  private items: StepDefinition<TValue>[];

  constructor(initial: StepDefinition<TValue>[]) {
    this.validateUniqueIds(initial);
    this.items = [...initial];
  }
}
```

### id 制約

- コレクション内の id は一意でなければならない
- 追加操作（`prepend` / `append` / `insertBefore` / `insertAfter`）で重複 id を検出した場合、即座に Error を throw する
- `insertBefore` / `insertAfter` で `refId` が存在しない場合も Error を throw する

### ミューテーション操作

```ts
class StepCollection<TValue> {
  /** 先頭に追加 */
  prepend(step: StepDefinition<TValue>): void {
    this.assertUniqueId(step.id);
    this.items.unshift(step);
  }

  /** 末尾に追加 */
  append(step: StepDefinition<TValue>): void {
    this.assertUniqueId(step.id);
    this.items.push(step);
  }

  /** 指定位置の前に挿入 */
  insertBefore(refId: string, step: StepDefinition<TValue>): void {
    this.assertUniqueId(step.id);
    const index = this.findIndexOrThrow(refId);
    this.items.splice(index, 0, step);
  }

  /** 指定位置の後に挿入 */
  insertAfter(refId: string, step: StepDefinition<TValue>): void {
    this.assertUniqueId(step.id);
    const index = this.findIndexOrThrow(refId);
    this.items.splice(index + 1, 0, step);
  }

  /** コレクションから step を削除する（step 数が減る） */
  removeStep(id: string): void {
    const index = this.findIndexOrThrow(id);
    this.items.splice(index, 1);
  }

  /** step の fn を差し替える（step 数は変わらない） */
  replaceFn(id: string, fn: StepMiddleware<TValue>): void {
    const index = this.findIndexOrThrow(id);
    this.items[index] = { ...this.items[index], fn };
  }

  /** step の fn を no-op（pass-through）に差し替える（step 数は変わらない） */
  disableStep(id: string): void {
    this.replaceFn(id, async (_ctx, next) => {
      await next();
    });
  }

  /** step の fn を wrap する（step 数は変わらない） */
  wrapStep(id: string, wrapper: StepMiddleware<TValue>): void {
    const index = this.findIndexOrThrow(id);
    const original = this.items[index].fn;
    this.items[index] = {
      ...this.items[index],
      fn: async (ctx, next) => {
        await wrapper(ctx, async () => {
          await original(ctx, next);
        });
      },
    };
  }

  has(id: string): boolean {
    return this.items.some((item) => item.id === id);
  }

  toArray(): readonly StepDefinition<TValue>[] {
    return [...this.items];
  }

  // --- private ---

  private assertUniqueId(id: string): void {
    if (this.has(id)) {
      throw new Error(`StepCollection: duplicate step id "${id}"`);
    }
  }

  private findIndexOrThrow(id: string): number {
    const index = this.items.findIndex((item) => item.id === id);
    if (index === -1) {
      throw new Error(`StepCollection: step id "${id}" not found`);
    }
    return index;
  }

  private validateUniqueIds(items: StepDefinition<TValue>[]): void {
    const seen = new Set<string>();
    for (const item of items) {
      if (seen.has(item.id)) {
        throw new Error(`StepCollection: duplicate step id "${item.id}"`);
      }
      seen.add(item.id);
    }
  }
}
```

### wrapStep の next チェーン

`wrapStep` は元の step 関数を wrapper で包む。
next の連鎖は二段構造になる。

```
パイプラインが wrappedFn(ctx, pipelineNext) を呼ぶ
  → wrapper(ctx, wrappedNext) が実行される
    → wrapper が前処理を行う
    → wrapper が next()（= wrappedNext）を呼ぶ
      → original(ctx, pipelineNext) が実行される
        → original が next()（= pipelineNext）を呼ぶ
          → パイプラインの次の step へ
        → original の後処理
    → wrapper の後処理
```

doc30 の例:

```ts
updatePrice.validation.wrap(async (ctx, next) => {
  // ← 前処理（権限チェック等）
  await next(); // ← ここで元の validation が実行される
  // ← 後処理
});
```

### 操作と step 数の対応

| 操作 | メソッド | step 数 |
|---|---|---|
| 先頭追加 | `prepend` | 増える |
| 末尾追加 | `append` | 増える |
| 指定位置前に挿入 | `insertBefore` | 増える |
| 指定位置後に挿入 | `insertAfter` | 増える |
| 削除 | `removeStep` | 減る |
| fn 差し替え | `replaceFn` | 変わらない |
| 無効化 | `disableStep` | 変わらない |
| wrap | `wrapStep` | 変わらない |

### 公開範囲

StepCollection は Framework の内部型であり、
消費者に公開される API は用途に応じて proxy 層で制限する。

| 消費者 | 公開される操作 | 制限の理由 |
|---|---|---|
| PresetTemplate | prepend / append / insertBefore / insertAfter / removeStep / replaceFn | テンプレートレベルの全操作（doc29 API一覧 4-2〜4-7） |
| ActionProxy.steps | prepend / append / insertBefore / insertAfter のみ | 個別 step の操作は StepHandle 経由（doc29 API一覧 5-12〜5-15） |
| StepHandle | replaceFn / wrapStep / disableStep | step id でアクセスした個別操作（doc29 API一覧 5-9〜5-11） |

## ResourceStore（.extend 対応）

doc28 の `createResourceStore` に `.extend()` メソッドを追加する。
Customize が既存 Store の state shape を拡張、
または既存フィールドの初期値を変更するために使う（doc29 API一覧 1-2, 1-3）。

### Immer middleware の組み込み（S10）

`createResourceStore` は Zustand の Immer middleware を内部で組み込む。
これにより:

- `getState()` が返す state は常に `Object.freeze` 済み
- state の直接 mutation（`state.price = 100`）はランタイムで TypeError になる
- スナップショットが step 中に汚染されることを構造的に防ぐ
- React コンポーネント内での accidental mutation も防止される

Core / Customize 開発者は Immer を意識する必要がない。
`ctx.store(X).set({ price: 100 })` / `ctx.store(X).update(...)` の API は変わらず、
内部で Immer が frozen state を管理する。

③ UI 状態（素の `create` で作る Store）は Immer の対象外である。

### 型定義

```ts
// framework/resourceStore.ts
import { create, StateCreator, StoreApi, UseBoundStore } from 'zustand';
import { immer } from 'zustand/middleware/immer';

export interface ResourceStoreMeta {
  readonly label: string;
}

export type ResourceStore<TState extends object> =
  UseBoundStore<StoreApi<TState>> & ResourceStoreMeta & {
    extend(partial: Partial<TState>): void;
  };
```

### createResourceStore

```ts
export const createResourceStore =
  <TState extends object>(label: string) =>
  (stateCreator: StateCreator<TState>): ResourceStore<TState> => {
    const store = create<TState>()(
      immer(stateCreator)
    ) as ResourceStore<TState>;

    Object.defineProperty(store, 'label', {
      value: label,
      enumerable: true,
      writable: false,
      configurable: false,
    });

    Object.defineProperty(store, 'extend', {
      value: (partial: Partial<TState>) => {
        store.setState(partial);
      },
      enumerable: true,
      writable: false,
      configurable: false,
    });

    return store;
  };
```

### extend の動作

- `store.setState(partial)` による shallow merge で指定フィールドのみ上書き
- 新規フィールドの追加と既存フィールドの初期値変更の両方に対応
- 型の拡張は TypeScript の module augmentation で行い、`extend` はランタイムの値注入のみを担う
- モジュール評価時に呼ばれることを前提とする

doc30 の例:

```ts
// 型の拡張（新規フィールド追加時のみ必要）
declare module '@core/store/useProductStore' {
  interface ProductStoreState {
    discountRate: number;
  }
}

// フィールド追加 + 初期値注入
useProductStore.extend({ discountRate: 0 });

// 既存フィールドの初期値変更（型変更不要）
useProductStore.extend({ price: 100 });
```

## MutableStoreGroup

doc28 の `defineStoreGroup` が返す StoreGroup に、
Customize 用のミューテーションメソッドを追加する（doc29 API一覧 2-2〜2-4）。

### 型定義

```ts
// framework/storeGroup.ts
import { ResourceStore } from '@core/infra/resourceStore';

export type StoreRef = ResourceStore<object> | StoreGroup;

export interface StoreGroup {
  readonly label: string;
  readonly refs: readonly StoreRef[];
  add(ref: StoreRef): void;
  remove(ref: StoreRef): void;
  replace(refs: readonly StoreRef[]): void;
}
```

### defineStoreGroup

```ts
export const defineStoreGroup = (
  label: string,
  initialRefs: readonly StoreRef[]
): StoreGroup => {
  const refs: StoreRef[] = [...initialRefs];

  const group: StoreGroup = {
    label,
    get refs() {
      return refs as readonly StoreRef[];
    },
    add(ref: StoreRef) {
      if (refs.includes(ref)) return;
      // 循環参照検出
      if (isStoreGroup(ref) && wouldCreateCycle(group, ref)) {
        throw new Error(
          `StoreGroup "${label}": adding "${ref.label}" would create a circular reference`
        );
      }
      refs.push(ref);
    },
    remove(ref: StoreRef) {
      const index = refs.indexOf(ref);
      if (index !== -1) {
        refs.splice(index, 1);
      }
    },
    replace(newRefs: readonly StoreRef[]) {
      // 循環参照検出
      for (const ref of newRefs) {
        if (isStoreGroup(ref) && wouldCreateCycle(group, ref)) {
          throw new Error(
            `StoreGroup "${label}": replacing with "${ref.label}" would create a circular reference`
          );
        }
      }
      refs.length = 0;
      refs.push(...newRefs);
    },
  };

  return group;
};
```

### 循環参照検出

`add` / `replace` 時に、追加先の StoreGroup の refs を再帰的に辿り、
自身に到達しないかを検査する。

```ts
function wouldCreateCycle(self: StoreGroup, candidate: StoreGroup): boolean {
  if (candidate === self) return true;
  for (const ref of candidate.refs) {
    if (isStoreGroup(ref) && wouldCreateCycle(self, ref)) {
      return true;
    }
  }
  return false;
}

function isStoreGroup(ref: StoreRef): ref is StoreGroup {
  return 'refs' in ref && Array.isArray((ref as StoreGroup).refs);
}
```

### 操作の振る舞い

| 操作 | 振る舞い |
|---|---|
| `add(ref)` | `refs` に追加。既に含まれていれば何もしない。循環参照なら throw |
| `remove(ref)` | `refs` から除去。見つからなければ何もしない |
| `replace(newRefs)` | `refs` の内容を全て入れ替える。循環参照なら throw |

一致判定は常にオブジェクト参照で行う（S7: label 文字列ではない）。

## StoreGroup 展開（flattenRefs）

Action scope に宣言された `StoreRef`（Store と StoreGroup の混在）を
再帰的に展開し、最終的な `ResourceStore` のフラット集合を得る。

S9 より、複数の StoreRef を参照するとき実効的な Store 集合は和集合で求める。

### flattenRefs

```ts
// framework/flattenRefs.ts

export function flattenRefs(
  refs: readonly StoreRef[]
): Set<ResourceStore<object>> {
  const result = new Set<ResourceStore<object>>();
  const visited = new Set<StoreGroup>();

  function walk(items: readonly StoreRef[]): void {
    for (const item of items) {
      if (isStoreGroup(item)) {
        if (visited.has(item)) continue;
        visited.add(item);
        walk(item.refs);
      } else {
        result.add(item);
      }
    }
  }

  walk(refs);
  return result;
}
```

### 振る舞い

- `ResourceStore` はそのまま結果集合に追加
- `StoreGroup` は `refs` を再帰的に展開
- 重複排除はオブジェクト参照で行う（`Set` の同一性判定）
- `visited` Set でフォールバックの循環防止を行う（主たる循環防止は `defineStoreGroup.add()` 時の検出）

## PresetTemplate

doc28 の preset（step 配列を返す関数）を、
Customize が操作可能なオブジェクトに変換する仕組みである（doc29 API一覧 4-2〜4-7）。

`definePreset` がこの仕組みを提供する。

### definePreset

```ts
// framework/presetTemplate.ts

export type PresetFactory<TParams, TValue> =
  (params: TParams) => StepDefinition<TValue>[];

export interface PresetTemplate<TParams, TValue> {
  /** 呼び出し: factory を実行し StepCollection を返す */
  (params: TParams): StepCollection<TValue>;

  /** テンプレートレベルのコレクション操作 */
  prepend(step: StepDefinition<TValue>): void;
  append(step: StepDefinition<TValue>): void;
  insertBefore(refId: string, step: StepDefinition<TValue>): void;
  insertAfter(refId: string, step: StepDefinition<TValue>): void;
  remove(id: string): void;
  replace(id: string, fn: StepMiddleware<TValue>): void;
}
```

### 実装

```ts
export function definePreset<TParams, TValue>(
  factory: PresetFactory<TParams, TValue>
): PresetTemplate<TParams, TValue> {
  type Mutation =
    | { type: 'prepend'; step: StepDefinition<TValue> }
    | { type: 'append'; step: StepDefinition<TValue> }
    | { type: 'insertBefore'; refId: string; step: StepDefinition<TValue> }
    | { type: 'insertAfter'; refId: string; step: StepDefinition<TValue> }
    | { type: 'remove'; id: string }
    | { type: 'replace'; id: string; fn: StepMiddleware<TValue> };

  const mutations: Mutation[] = [];
  const instances: StepCollection<TValue>[] = [];

  function applyMutation(
    collection: StepCollection<TValue>,
    mutation: Mutation
  ): void {
    switch (mutation.type) {
      case 'prepend':
        collection.prepend(mutation.step);
        break;
      case 'append':
        collection.append(mutation.step);
        break;
      case 'insertBefore':
        collection.insertBefore(mutation.refId, mutation.step);
        break;
      case 'insertAfter':
        collection.insertAfter(mutation.refId, mutation.step);
        break;
      case 'remove':
        collection.removeStep(mutation.id);
        break;
      case 'replace':
        collection.replaceFn(mutation.id, mutation.fn);
        break;
    }
  }

  function addMutation(mutation: Mutation): void {
    // 既存インスタンスへの適用を先に行う。
    // 一つでも throw すれば mutations には入らない（定義の汚染防止）。
    for (const instance of instances) {
      applyMutation(instance, mutation);
    }
    mutations.push(mutation);
  }

  const template = function (params: TParams): StepCollection<TValue> {
    const baseSteps = factory(params);
    const collection = new StepCollection<TValue>(baseSteps);

    // 蓄積済みテンプレートミューテーションを適用
    for (const mutation of mutations) {
      applyMutation(collection, mutation);
    }

    instances.push(collection);
    return collection;
  } as PresetTemplate<TParams, TValue>;

  template.prepend = (step) => addMutation({ type: 'prepend', step });
  template.append = (step) => addMutation({ type: 'append', step });
  template.insertBefore = (refId, step) =>
    addMutation({ type: 'insertBefore', refId, step });
  template.insertAfter = (refId, step) =>
    addMutation({ type: 'insertAfter', refId, step });
  template.remove = (id) => addMutation({ type: 'remove', id });
  template.replace = (id, fn) => addMutation({ type: 'replace', id, fn });

  return template;
}
```

### Core 側の書き方の変更

doc28 の preset 定義は `definePreset` でラップする。

```ts
// core/presets/fieldInput.ts（doc28 からの変更）
import { definePreset } from '@framework/presetTemplate';
import { sanitize } from '@core/middlewares/sanitize';
import { zodValidation } from '@core/middlewares/zodValidation';

export const fieldInputPreset = definePreset(({ validation, logic, notify }) => [
  { id: 'sanitize', fn: sanitize },
  { id: 'validation', fn: zodValidation(validation) },
  { id: 'logic', fn: logic },
  { id: 'notify', fn: notify },
]);
```

呼び出し側は変わらない。`fieldInputPreset({ ... })` で `StepCollection` が返る。
`actionRegistry.register` は `StepCollection` と `StepDefinition[]` の両方を受け付ける。

### テンプレートレベル変更の伝播

テンプレートレベルの操作は、
既に生成された全インスタンスと、今後生成されるインスタンスの両方に適用される。
これにより doc30 の「テンプレートレベルの変更であり、その preset を使う全 Action に波及する」が実現される。

```
時刻1: const stepsA = fieldInputPreset({ ... })
        → [sanitize, validation, logicA, notifyA]
時刻2: const stepsB = fieldInputPreset({ ... })
        → [sanitize, validation, logicB, notifyB]
時刻3: fieldInputPreset.prepend({ id: 'preCheck', fn })
        → stepsA: [preCheck, sanitize, validation, logicA, notifyA]
        → stepsB: [preCheck, sanitize, validation, logicB, notifyB]
時刻4: const stepsC = fieldInputPreset({ ... })
        → [preCheck, sanitize, validation, logicC, notifyC]
```

テンプレートレベル操作と Action レベル操作は同一の StepCollection に対して行われる。
両者はモジュール評価順に適用されるため、順序は決定的である。

## ActionProxy

ActionProxy は `actionRegistry.register` が返すオブジェクトであり、
Action の定義を保持し、Customize からの全操作を受け付ける。

### 内部状態（ActionDefinition）

```ts
// framework/actionProxy.ts

interface ActionDefinition<TValue> {
  id: string;
  scope: {
    internalStores: StoreRef[];
    calls: (ActionProxy<any> | (() => ActionProxy<any>))[];
  };
  steps: StepCollection<TValue>;
  on: Set<string>;
  bindings: BindingRegistry;
  disabled: boolean;
}
```

`scope.calls` は `ctx.call` で呼び出す子 Action の配列。thunk（`() => ActionProxy`）も受け付ける（ES module の forward reference を解決するため）。省略時は空配列で初期化する。排他 scope の推移閉包計算に使われる（S11）。

### ActionProxy の概念型

```ts
export interface ActionProxy<TValue> {
  /** 呼び出し: パイプラインを起動する */
  (value: TValue): Promise<ActionResult>;

  /** Action id（読み取り専用） */
  readonly id: string;

  /** step を id でアクセスする（動的プロパティ） */
  readonly [stepId: string]: StepHandle<TValue>;

  /** steps コレクション操作 */
  readonly steps: StepsProxy<TValue>;

  /** scope 操作 */
  readonly scope: ScopeProxy;

  /** シグナル購読操作 */
  readonly on: SignalSetProxy;

  /** 全体差し替え */
  replace(definition: {
    scope: ActionScope;
    steps: StepCollection<TValue> | StepDefinition<TValue>[];
    on?: string[];
  }): void;

  /** 無効化 */
  disable(): void;

  /** id 変更 */
  setId(newId: string): void;

  /** bindContext 登録 */
  bindContext(bindingId: string, options: BindingOptions): void;

  /** bindContext 削除 */
  removeBindContext(bindingId: string): void;
}
```

注: 上記は概念型であり、TypeScript のインデックスシグネチャと固定プロパティの型衝突がある。
実際の実装は JS Proxy で動的プロパティアクセスを解決する。

step id として使うことはできない。

```ts
const RESERVED_PROPERTIES = new Set([
```

### JS Proxy による実装

```ts
/** useActionStatus が ActionProxy から statusStore を取得するための Symbol キー */
export const STATUS_STORE = Symbol('actionStatusStore');

/** executeChildCall が ActionProxy から ActionDefinition を取得するための Symbol キー */
export const DEFINITION = Symbol('actionDefinition');

function createActionProxy<TValue>(
  definition: ActionDefinition<TValue>
): ActionProxy<TValue> {
  // Action ごとの状態管理用ミニストア（useActionStatus の内部ストア）
  const statusStore = createStore<ActionStatus>(() => ({ phase: 'idle' as const, error: null }));

  let proxyRef: ActionProxy<TValue>;

  const invoke = (value: TValue): Promise<ActionResult> => {
    if (definition.disabled) {
      return Promise.resolve({ ok: true });
    }
    return runPipeline(proxyRef, definition, value, statusStore);
  };

  const proxy = new Proxy(invoke, {
    get(_target, prop: string | symbol) {
      if (prop === STATUS_STORE) return statusStore;
      if (prop === DEFINITION) return definition;
      if (typeof prop === 'symbol') return undefined;

      switch (prop) {
        case 'id':
          return definition.id;
        case 'steps':
          return createStepsProxy(definition.steps);
        case 'scope':
          return createScopeProxy(definition.scope);
        case 'on':
          return createSignalSetProxy(definition.on);
        case 'replace':
          return replaceAction;
        case 'disable':
          return () => { definition.disabled = true; };
        case 'setId':
          return (newId: string) => { definition.id = newId; };
        case 'bindContext':
          return (bindingId: string, options: BindingOptions) => {
            definition.bindings.bind(bindingId, options);
          };
        case 'removeBindContext':
          return (bindingId: string) => {
            definition.bindings.unbind(bindingId);
          };
        default:
          // 動的 step アクセス
          if (definition.steps.has(prop)) {
            return createStepHandle(definition.steps, prop);
          }
          return undefined;
      }
    },
    apply(_target, _thisArg, args) {
      return invoke(args[0] as TValue);
    },
  }) as ActionProxy<TValue>;
  proxyRef = proxy;

  function replaceAction(newDef: {
    scope: ActionScope;
    steps: StepCollection<TValue> | StepDefinition<TValue>[];
    on?: string[];
  }): void {
    definition.scope = {
      internalStores: [...newDef.scope.internalStores],
      calls: [...(newDef.scope.calls ?? [])],
    };
    definition.steps =
      newDef.steps instanceof StepCollection
        ? newDef.steps
        : new StepCollection(newDef.steps);
    definition.on = new Set(newDef.on ?? []);
    definition.bindings = new BindingRegistry();
    definition.disabled = false;
  }

  return proxy;
}
```

Proxy の外殻（関数オブジェクト自体）は変わらないため、
既存の import 参照は維持される（S4）。

### StepHandle

step を id でアクセスしたときに返るオブジェクト。
呼び出し可能（fn 差し替え）であり、`.wrap()` と `.remove()` を持つ。

```ts
interface StepHandle<TValue> {
  /** 呼び出し: step の fn を差し替える（step 数は変わらない） */
  (fn: StepMiddleware<TValue>): void;
  /** 元の step を wrap する（step 数は変わらない） */
  wrap(fn: StepMiddleware<TValue>): void;
  /** step を無効化する（fn を pass-through に差し替え、step 数は変わらない） */
  remove(): void;
}
```

```ts
function createStepHandle<TValue>(
  collection: StepCollection<TValue>,
  stepId: string
): StepHandle<TValue> {
  const handle = function (fn: StepMiddleware<TValue>) {
    collection.replaceFn(stepId, fn);
  } as StepHandle<TValue>;

  handle.wrap = (fn: StepMiddleware<TValue>) => {
    collection.wrapStep(stepId, fn);
  };

  handle.remove = () => {
    collection.disableStep(stepId);
  };

  return handle;
}
```

doc30 の例:

```ts
// ロジック差し替え（呼び出し）
updatePrice.logic(async (ctx, next) => { /* ... */ await next(); });

// wrap（前後に処理追加）
updatePrice.validation.wrap(async (ctx, next) => {
  // 前処理
  await next(); // ← 元の validation が実行される
  // 後処理
});

// 無効化
updatePrice.notify.remove();
```

### StepsProxy

ActionProxy の `.steps` プロパティ。
コレクションの挿入操作のみを公開する。

```ts
interface StepsProxy<TValue> {
  prepend(step: StepDefinition<TValue>): void;
  append(step: StepDefinition<TValue>): void;
  insertBefore(refId: string, step: StepDefinition<TValue>): void;
  insertAfter(refId: string, step: StepDefinition<TValue>): void;
}
```

```ts
function createStepsProxy<TValue>(
  collection: StepCollection<TValue>
): StepsProxy<TValue> {
  return {
    prepend: (step) => collection.prepend(step),
    append: (step) => collection.append(step),
    insertBefore: (refId, step) => collection.insertBefore(refId, step),
```

### ScopeProxy

ActionProxy の `.scope` プロパティ。
scope のミューテーションを提供する。

```ts
interface ScopeCollectionProxy<T = StoreRef> {
  add(ref: T): void;
  remove(ref: T): void;
}

type CallsRef = ActionProxy<any> | (() => ActionProxy<any>);

interface ScopeProxy {
  readonly internalStores: ScopeCollectionProxy;
  readonly calls: ScopeCollectionProxy<CallsRef>;
}
```

```ts
function createScopeProxy(
  scope: ActionDefinition<any>['scope']
): ScopeProxy {
  function createCollectionProxy<T>(
    collection: T[]
  ): ScopeCollectionProxy<T> {
    return {
      add(ref: T) {
        if (!collection.includes(ref)) {
          collection.push(ref);
        }
      },
      remove(ref: T) {
        const index = collection.indexOf(ref);
        if (index !== -1) {
          collection.splice(index, 1);
        }
      },
    };
  }

  return {
    internalStores: createCollectionProxy(scope.internalStores),
    calls: createCollectionProxy(scope.calls),
  };
}
```

doc30 の例:

```ts
// scope の拡張（内部世界の追加）
updatePrice.scope.internalStores.add(useMemberStore);

// calls への追加（子 Action の宣言）
updatePrice.scope.calls.add(recalculateTaxPreview);

// scope の削減
updatePrice.scope.internalStores.remove(store);
```

### computeEffectiveScope（排他 scope の推移閉包）

排他 scope は Action 自身の `internalStores` に加え、`calls` で宣言された全子孫の `internalStores` を再帰的に合算した推移閉包である（S11）。

この計算は Action 実行のたびに行う（キャッシュしない）。
scope の変更（`ScopeProxy` の `add` / `remove`、`StoreGroup` の `add` / `remove` / `replace`）が
任意のタイミングで発生しうるため、キャッシュの invalidation を管理するより毎回計算するほうが単純かつ安全である。
計算コストは現実的な Action 構成（Store 数十個、calls 深さ数段）ではマイクロ秒オーダーであり、
step 実行や API 呼び出しのコストに対して無視できる。

thunk（`() => ActionProxy`）は計算時に解決される（ES module の forward reference を回避するため）。

```ts
// framework/effectiveScope.ts

function computeEffectiveScope(
  action: ActionProxy<any>,
  visiting: Set<ActionProxy<any>> = new Set()
): Set<ResourceStore<object>> {
  // 循環検出
  if (visiting.has(action)) {
    throw new Error(
      `Circular calls detected: ${[...visiting].map(a => a.id).join(' → ')} → ${action.id}`
    );
  }
  visiting.add(action);

  // DEFINITION Symbol 経由で生の定義を取得（ScopeProxy はイテレーション不可のため）
  const def = (action as any)[DEFINITION] as ActionDefinition<any>;

  // 自身の internalStores を展開（StoreGroup → 個別 Store）
  const stores = new Set(flattenRefs(def.scope.internalStores));

  // calls の推移閉包
  for (const callRef of def.scope.calls) {
    const child = (callRef as any)[DEFINITION] !== undefined
      ? callRef as ActionProxy<any>
      : (callRef as () => ActionProxy<any>)();
    const childScope = computeEffectiveScope(child, new Set(visiting));
    for (const store of childScope) {
      stores.add(store);
    }
  }

  visiting.delete(action);
  return stores;
}
```

- `visiting` は循環検出用。`calls` が A → B → A のように循環する場合はエラー
- `flattenRefs` は `StoreRef[]` から StoreGroup を展開し、フラットな Store 集合を得るヘルパー（後述の `runPipeline` でも使用）

### SignalSetProxy

ActionProxy の `.on` プロパティ。
シグナル購読の集合操作を提供する（doc29 API一覧 5-16〜5-18）。

```ts
interface SignalSetProxy {
  add(signalId: string): void;
  remove(signalId: string): void;
  replace(oldSignalId: string, newSignalId: string): void;
}
```

```ts
function createSignalSetProxy(signals: Set<string>): SignalSetProxy {
  return {
    add(signalId: string) {
      signals.add(signalId);
    },
    remove(signalId: string) {
      signals.delete(signalId);
    },
    replace(oldSignalId: string, newSignalId: string) {
      signals.delete(oldSignalId);
      signals.add(newSignalId);
    },
  };
}
```

### 全体差し替え（replace）

ActionDefinition の全フィールドを新しい定義で上書きする。
Proxy の外殻は変わらないため import 参照は維持される。
bindings もリセットされる（「Core の定義を完全に捨てて書き直す最も重い手段」の解釈）。

### 無効化（disable）

`definition.disabled = true` を設定する。

無効化された Action の振る舞い:

| 起動方法 | 振る舞い |
|---|---|
| 直接呼び出し `updatePrice(value)` | パイプラインを起動せず `{ ok: true }` を返す |
| `ctx.call` 経由 | savepoint を取らず即座に return する（no-op）。親の step は正常に続行する |
| シグナル経由 `on` 購読 | `findSubscribers` がスキップする。Action は呼ばれない |

無効化はグローバル操作である。全呼び出し元（直接呼び出し・ctx.call・シグナル）で no-op になる。
`customize/globals/` に置く（doc29 5-3）。

無効化中も `on.add()` / `on.remove()` 等のミューテーション操作は許可する。
re-enable の手段は `replace` による再定義。

### id 変更（setId）

`definition.id = newId` を設定する。
registry Map のキーは更新しない（registry のキーは登録時の id で固定）。
Action の参照解決は JS import（オブジェクト参照）で行うため、id の変更はエラーメッセージ・デバッグ表示にのみ影響する。

## BindingRegistry（bindContext）

Action ごとの bindContext 登録を管理し、
ctx 構築時に解決する（doc29 API一覧 5-19〜5-20）。

### 内部構造

```ts
// framework/bindingRegistry.ts

  readonly bindingId: string;
  readonly keys: readonly string[];

}

class BindingRegistry {
```

### BindingOptions

```ts

  keys: string[];

}
```

### 予約キー

以下のキーは `keys` に含めることを禁止する。
登録時に即座に Error を throw する。

```ts
const RESERVED_CONTEXT_KEYS = new Set([
  'value',
  'store', 'emit', 'call',
  'getRuntime', 'setRuntime', 'updateRuntime',
]);
```

### 登録（bind）

```ts
bind(bindingId: string, options: BindingOptions): void {
  // 予約キーチェック
  for (const key of options.keys) {
    if (RESERVED_CONTEXT_KEYS.has(key)) {
      throw new Error(
        `bindContext: "${key}" is a reserved context key`
      );
    }
  }

  // 第1層: 同一 bindingId の既存エントリを削除（後勝ち）
  this.entries = this.entries.filter(
    (e) => e.bindingId !== bindingId
  );

  // 末尾に追加
  this.entries.push({
    bindingId,
    keys: options.keys,
    factory: options.factory,
  });
}
```

### 削除（unbind）

```ts
unbind(bindingId: string): void {
  this.entries = this.entries.filter(
    (e) => e.bindingId !== bindingId
  );
}
```

### ctx 構築時の解決（resolve）

パイプライン開始前に呼ばれる。
全エントリの factory を実行し、結果をマージして返す。

```ts
resolve(): Record<string, unknown> {
  const result: Record<string, unknown> = {};

  for (const entry of this.entries) {
    const payload = entry.factory();

    if (process.env.NODE_ENV === 'development') {
      // 不足チェック（keys にあるが factory が返さない）
      for (const key of entry.keys) {
        if (!(key in payload)) {
          throw new Error(
            `bindContext "${entry.bindingId}": ` +
            `key "${key}" declared in keys but not returned by factory`
          );
        }
      }
      // 超過チェック（keys にないが factory が返す）
      for (const key of Object.keys(payload)) {
        if (!entry.keys.includes(key)) {
          throw new Error(
            `bindContext "${entry.bindingId}": ` +
            `factory returned undeclared key "${key}"`
          );
        }
      }
    }

    // keys に宣言されたもののみマージ（第2層: 後勝ち）
    for (const key of entry.keys) {
      if (key in payload) {
        result[key] = payload[key];
      } else if (process.env.NODE_ENV !== 'development') {
        // 本番: undefined として扱い警告ログ
        console.warn(
          `bindContext "${entry.bindingId}": ` +
          `key "${key}" missing from factory result`
        );
        result[key] = undefined;
      }
    }
  }

  return result;
}
```

### 二層衝突ルール

| 層 | 単位 | ルール | 実装 |
|---|---|---|---|
| 第1層 | `bindingId` | 同じ `bindingId` で再登録 → 既存を削除して末尾に追加 | `bind` 内の filter + push |
| 第2層 | payload key | 異なる `bindingId` から同じキー → 後に登録された factory の値を採用 | `resolve` 内の順次上書き |

`entries` 配列の順序（= 登録順）に従って順次マージするため、
後勝ちは自然に実現される。

### 開発 / 本番の振る舞い差異

| 環境 | 不足（keys にあるが factory が返さない） | 超過（keys にないが factory が返す） |
|---|---|---|
| 開発 | 即 throw | 即 throw |
| 本番 | `undefined` として扱い、警告ログ | `ctx` にマージせず除外 |

doc30 の制約:

- factory は同期・軽量・無副作用（Promise を返してはならない）
- モジュール評価時に登録（React Component 内など動的タイミングでの登録は禁止）
- 注入された値は Action 実行中に再代入しない

## RuntimeMap

ステップ間の一時データを保持する `Map<string, unknown>` ラッパー。
doc28 の `ctx.getRuntime / ctx.setRuntime / ctx.updateRuntime` に対応する。

```ts
// framework/runtimeMap.ts

class RuntimeMap {
  private data = new Map<string, unknown>();

  get(key: string): unknown {
    return this.data.get(key);
  }

  set(key: string, value: unknown): void {
    this.data.set(key, value);
  }

  update(key: string, updater: (current: unknown) => unknown): unknown {
    const updated = updater(this.data.get(key));
    this.data.set(key, updated);
    return updated;
  }
}
```

パイプライン開始時に空の `RuntimeMap` を生成し、`ctx` に注入する:

```ts
const runtimeMap = new RuntimeMap();

// ActionContext 構築時

setRuntime: (key: string, value: unknown) => runtimeMap.set(key, value),

  runtimeMap.update(key, updater),
```

- RuntimeMap はパイプラインごとに生成され、パイプライン終了時に破棄される
- `ctx.call` で子 Action を呼んだ場合、子は自身の RuntimeMap を持つ（親とは共有しない）
- 型安全性は提供しない（`unknown`）。消費側で `as` キャストする（doc28:522 参照）

## actionRegistry

Action の登録と管理を行うグローバルレジストリ。

### 実装

```ts
// framework/actionRegistry.ts

interface RegistryEntry {
  proxy: ActionProxy<any>;
  definition: ActionDefinition<any>;
}

const registry = new Map<string, RegistryEntry>();

export const actionRegistry = {
  register<TValue>(definition: {
    id: string;
    scope: ActionScope;
    steps: StepCollection<TValue> | StepDefinition<TValue>[];
    on?: string[];
  }): ActionProxy<TValue> {
    const actionDef: ActionDefinition<TValue> = {
      id: definition.id,
      scope: {
        internalStores: [...definition.scope.internalStores],
        calls: [...(definition.scope.calls ?? [])],
      },
      steps:
        definition.steps instanceof StepCollection
          ? definition.steps
          : new StepCollection(definition.steps),
      on: new Set(definition.on ?? []),
      bindings: new BindingRegistry(),
      disabled: false,
    };

    if (registry.has(definition.id)) {
      throw new Error(
        `actionRegistry.register: id "${definition.id}" は既に登録されています`
      );
    }

    const proxy = createActionProxy(actionDef);
    registry.set(definition.id, { proxy, definition: actionDef });
    return proxy;
  },
};
```

### register の振る舞い

1. `id` の重複チェック。既に同じ id が登録されていれば throw する
2. `scope.internalStores` を内部配列にコピーする
3. `scope.calls` を内部配列にコピーする（`undefined` なら空配列）
4. `steps` が `StepCollection` ならそのまま参照する（preset 経由の場合）。配列なら `new StepCollection()` でラップする
5. `on` を `Set<string>` に変換する
6. `BindingRegistry` を空で初期化する
7. `ActionDefinition` を生成し `createActionProxy` で Proxy を作る
8. `registry` Map に `id` をキーとして登録する
9. Proxy を返す

`steps` が `StepCollection` の場合にそのまま参照するのは、
PresetTemplate のテンプレートレベル変更を伝播させるためである。
コピーすると伝播が切れる。

## パイプライン実行

`framework/runner.ts`

doc28 のエラーハンドリング節（pipeline runner の処理順序）に対応する。

### 確定済みの判断

- **スナップショット方式**: Immer middleware により state は常に frozen。`getState()` の返却値がそのままスナップショットになる。frozen オブジェクトは step 中に mutation されないため、追加のコピーは不要
- **バッファ合成結果の deep freeze**: `BufferedStoreHandle.get()` がバッファとスナップショットをマージして返す場合、Immer の `freeze(obj, true)` で deep freeze する。スプレッドで生成した新規オブジェクトは unfrozen であり、freeze しなければ直接代入が黙って成功するがバッファには反映されない（RW 整合性の破壊）。shallow freeze ではネストされた buffer 由来のプロパティが unfrozen のまま残るため、deep freeze が必要。S10 の保護境界をバッファモデル内でも維持するために必須
- **step チェーンの実行方式**: Koa-style compose。index ガードにより next() の多重呼び出しを検出してエラーにする

```ts
function compose<TValue>(
  steps: readonly StepDefinition<TValue>[],
) {
  return function (ctx: ActionContext<TValue>): Promise<void> {
    let index = -1;
    function dispatch(i: number): Promise<void> {
      if (i <= index) {
        return Promise.reject(new Error('next() called multiple times'));
      }
      index = i;
      if (i >= steps.length) return Promise.resolve();
      return Promise.resolve(steps[i].fn(ctx, () => dispatch(i + 1)));
    }
    return dispatch(0);
  };
}
```

- **ctx.store() の書き込みパス**: 内部バッファ + 成功時一括 commit。パイプライン中は Zustand の state を変更せず、バッファ上で読み書きする。全 step 成功後にバッファを Zustand に commit する。失敗時はバッファ破棄のみ（Zustand 未変更のためロールバック操作不要）。doc28 の「即時反映」は「パイプライン内で `get()` すれば直ちに最新値が読める」として実現する

#### StoreHandle（internalStores 用）

```ts
class BufferedStoreHandle<TState extends object> implements StoreHandle<TState> {
  private readonly snapshot: Readonly<TState>; // Immer frozen — コピー不要
  private buffer: Partial<TState> = {};
  private readonly store: ResourceStore<TState>;

  constructor(store: ResourceStore<TState>, snapshotOverride?: Readonly<TState>) {
    this.store = store;
    this.snapshot = snapshotOverride ?? store.getState();
  }

  get(): Readonly<TState> {
    if (Object.keys(this.buffer).length === 0) return this.snapshot;
    return freeze({ ...this.snapshot, ...this.buffer }, true) as Readonly<TState>;
  }

  set(partial: Partial<TState>): void {
    Object.assign(this.buffer, partial);
  }

  update(updater: (current: Readonly<TState>) => Partial<TState>): void {
    Object.assign(this.buffer, updater(this.get()));
  }

  /** バッファの内容を返す（savepoint のスナップショット用。Framework 内部専用） */
  getBuffer(): Partial<TState> {
    return this.buffer;
  }

  /** バッファを指定したスナップショットの状態に戻す（savepoint の復元用。Framework 内部専用） */
  restoreBuffer(snapshot: Partial<TState>): void {
    this.buffer = snapshot;
  }

  /** バッファをクリアし、snapshot 状態に戻す */
  reset(): void {
    this.buffer = {};
  }

  /** 全 step 成功後に呼ぶ: バッファを Zustand に反映 */
  commit(): void {
    if (Object.keys(this.buffer).length > 0) {
      this.store.setState(this.buffer);
    }
  }

}
```

- `snapshotOverride` は Zustand の `getState()` 以外の初期値でハンドルを生成する場合に使��（将来の拡張点として残す）
- `get()` はバッファが空なら snapshot をそのまま返す（マージコストなし）
- `set()` / `update()` はバッファに蓄積するだけで Zustand を触らない
- `reset()` はバッファを空にして snapshot 状態に戻す。特定 Store だけ書き込みを撤回したい場合に使う
- `getBuffer()` はバッファの生値を返す。savepoint のスナップショット取得に使う。Framework 内部専用
- `restoreBuffer(snapshot)` はバッファを指定したスナップショットの状態に戻す。savepoint の復元に使う。Framework 内部専用
- `commit()` は全 step 成功後に runner が呼ぶ。Zustand の setState を 1 回だけ実行

#### パイプライン実行フロー（バッファモデル）

doc28 の pipeline runner 処理順序をバッファモデルで実現する。

```
1. 排他 scope（推移閉包）を計算し、排他ロックを取得する（重なる Action が実行中なら待機）
2. statusStore を { phase: 'running' } に更新
3. 排他 scope の snapshot を取得（getState() — S10 により frozen）
4. 各 Store に対して BufferedStoreHandle を生成（snapshot + buffer、get/set/update/reset）
5. steps を compose で実行（バッファ上で読み書き。Zustand は不変）
6. step 失敗 → 全バッファ破棄（Zustand 未変更）
   → statusStore を { phase: 'done', error } に更新 → 排他ロック解放 → { ok: false, error }
7. 全 step 成功 → StoreHandle.commit()（全 Store を Zustand に反映）→ snapshot 解放
8. emitQueue をディスパッチ（購読 Action を独立 top-level Action として逐次実行）
   購読 Action の失敗は emitter の結果に影響しない
   → statusStore を { phase: 'done', error: null } に更新 → 排他ロック解放 → { ok: true }
```

```ts
// runner（runPipeline）の概略
async function runPipeline<TValue>(
  proxy: ActionProxy<TValue>,
  definition: ActionDefinition<TValue>,
  value: TValue,
  statusStore: StoreApi<ActionStatus>,
): Promise<ActionResult> {
  // 1. 排他ロック取得（重なる Action が実行中なら待機）
  const effectiveStores = computeEffectiveScope(proxy);
  await exclusionManager.acquire(effectiveStores);

  try {
  // 2. running
  statusStore.setState({ phase: 'running', error: null });

  // 3-4. 排他 scope でバッファ確保 + ハンドル生成 + ctx 構築

  const pipeline: PipelineContext = {
    storeHandles: new Map(),
    savepointStack: [],
    emitQueue: [],
  };

  for (const store of effectiveStores) {
    pipeline.storeHandles.set(store, new BufferedStoreHandle(store));
  }

  const runtimeMap = new RuntimeMap();

  const ctx: ActionContext<TValue> = {
    value,

    store<T extends object>(ref: ResourceStore<T>): StoreHandle<T> {
      const handle = pipeline.storeHandles.get(ref);
      if (!handle) {
        throw new Error(
          `Action "${definition.id}": ctx.store() に渡された "${ref.label}" は ` +
          `排他 scope（推移閉包）に含まれていません`
        );
      }
      return handle as StoreHandle<T>;
    },

    emit(signalId: string, payload?: unknown): void {
      pipeline.emitQueue.push({ signalId, payload });
    },

    async call<C>(childAction: ActionProxy<C>, childValue: C): Promise<void> {
      // calls 違反検出 + savepoint + 子 steps 実行（「ctx.call と savepoint」節参照）
      await executeChildCall(definition, pipeline, childAction, childValue);
    },

    getRuntime: (key: string) => runtimeMap.get(key),
    setRuntime: (key: string, v: unknown) => runtimeMap.set(key, v),
    updateRuntime: (key: string, updater: (current: unknown) => unknown) =>
      runtimeMap.update(key, updater),

    // bindContext の解決結果をスプレッド
    ...definition.bindings.resolve(),
  };

  const composedSteps = compose(definition.steps.toArray());

  // 5. step 実行
  try {
    await composedSteps(ctx);
  } catch (error) {
    // 6. step 失敗
    statusStore.setState({ phase: 'done', error });
    return { ok: false, error };
  }

  // 7. commit（全 Store を Zustand に反映）+ snapshot 解放
  for (const handle of pipeline.storeHandles.values()) handle.commit();

  // 8. emit ディスパッチ（commit 後に独立 Action として実行）
  await dispatchEmitQueue(pipeline.emitQueue);

  statusStore.setState({ phase: 'done', error: null });
  return { ok: true };
  } finally {
    // 排他ロック解放（成功・失敗を問わず必ず解放）
    exclusionManager.release(effectiveStores);
  }
}
```

React への影響:

| フェーズ | Zustand state | statusStore | React 描画 |
|---|---|---|---|
| 排他取得待ち（1） | 不変 | 変化なし | なし |
| running（2） | 不変 | `{ phase: 'running' }` | statusStore 購読者のみ |
| step 実行中（5） | 不変 | 変化なし | なし |
| step 失敗（6） | 不変 | `{ phase: 'done', error }` | statusStore 購読者のみ |
| commit（7） | 更新 | 変化なし | 1 回 |
| emit ディスパッチ（8） | 購読 Action 依存 | 変化なし | 購読 Action 依存 |
| 完了 | — | `{ phase: 'done', error: null }` | statusStore 購読者 |

## 排他制御

doc27 の排他制御仕様（悲観ロック、排他 scope 単位）を実装する。

### 排他判定

二つの Action の排他 scope（`computeEffectiveScope` の結果）に共通する Store が 1 つでもあれば、排他が必要である（doc27:94）。

### ExclusionManager

排他ロックの取得・解放・待機を管理するシングルトン。

```ts
// framework/exclusionManager.ts

type Resolve = () => void;

interface RunningEntry {
  stores: Set<ResourceStore<object>>;
}

class ExclusionManager {
  private running: RunningEntry[] = [];
  private queue: { stores: Set<ResourceStore<object>>; resolve: Resolve }[] = [];

  async acquire(stores: Set<ResourceStore<object>>): Promise<void> {
    if (this.hasConflict(stores)) {
      // 重なる Action が実行中 → キューに入れて待機
      return new Promise<void>((resolve) => {
        this.queue.push({ stores, resolve });
      });
    }
    this.running.push({ stores });
  }

  release(stores: Set<ResourceStore<object>>): void {
    // 実行中リストから除去
    const index = this.running.findIndex((entry) => entry.stores === stores);
    if (index !== -1) this.running.splice(index, 1);

    // キューから実行可能になったものを起こす
    this.flush();
  }

  private hasConflict(stores: Set<ResourceStore<object>>): boolean {
    return this.running.some((entry) => this.overlaps(entry.stores, stores));
  }

  private overlaps(a: Set<ResourceStore<object>>, b: Set<ResourceStore<object>>): boolean {
    for (const store of a) {
      if (b.has(store)) return true;
    }
    return false;
  }

  private flush(): void {
    let i = 0;
    while (i < this.queue.length) {
      const entry = this.queue[i];
      if (!this.hasConflict(entry.stores)) {
        this.queue.splice(i, 1);
        this.running.push({ stores: entry.stores });
        entry.resolve();
      } else {
        i++;
      }
    }
  }
}

export const exclusionManager = new ExclusionManager();
```

### 動作

1. `acquire(stores)` — 実行中リストと scope の重なりを判定する。重なりがなければ実行中リストに追加して即 resolve。重なりがあればキューに入れて Promise で待機する
2. `release(stores)` — 実行中リストから除去し、`flush()` でキュー内の待機中エントリを先頭から順に判定する。重なりがなくなったエントリを実行中リストに移して resolve する
3. `flush()` — キューを先頭から走査し、実行可能なエントリを全て起こす。複数の待機 Action が同時に解放される場合がある（互いに scope が重ならない場合）

### デッドロック防止

排他 scope は推移閉包で Action 開始前に確定する（doc27:134-138）。
`acquire` は排他 scope 全体を一括で取得する（段階的なリソース取得はしない）。
リソースの段階的取得がなければ、循環待ちによるデッドロックは構造的に起きない。

### ctx.call と排他制御の関係

`ctx.call` で呼ばれる子 Action は、親の排他ロック内で実行される。
子の排他 scope は親の排他 scope の部分集合であるため（推移閉包の定義より）、
子は追加のロック取得を必要としない。`executeChildCall` は `exclusionManager` を呼ばない。

### シグナルディスパッチと排他制御の関係

`dispatchEmitQueue` で実行される購読 Action は独立した top-level Action として実行される。
各購読 Action は自身の `runPipeline` を通じて独自に `acquire` / `release` する。
emitter の排他ロックは emit ディスパッチ後に解放されるため（ステップ 8 の後の finally）、
購読 Action と emitter の排他 scope が重なる場合、購読 Action は emitter の解放を待つ。

## useActionStatus

`framework/useActionStatus.ts`

Action の実行状態を React コンポーネントから購読するための hook。
doc28 の使用例（`useActionStatus(updatePrice)` → `status.error instanceof ValidationError`）を実現する。

### ActionStatus 型

```ts
type ActionStatus =
  | { phase: 'idle'; error: null }
  | { phase: 'running'; error: null }
  | { phase: 'done'; error: Error | null };
```

- `phase: 'idle'` — 未実行（初期状態）
- `phase: 'running'` — パイプライン実行中
- `phase: 'done'` — 完了。`error: null` なら成功、`error: Error` なら失敗

全 phase に `error` フィールドを持たせることで、doc28 のように phase を確認せず `status.error instanceof ValidationError` と書ける。`idle` / `running` では常に `null`。

### 内部ストア

Action ごとに Zustand vanilla store（`createStore`）を生成する。
`createResourceStore`（業務 Store 用、Immer middleware 付き）ではなく素の `createStore` を使う。理由:

- statusStore は Framework 内部の関心であり、業務データではない
- Immer の frozen 保護は不要（runner だけが setState する）
- Customize からの `.extend()` も不要

生成と保持は `createActionProxy` 内の closure（ActionProxy 節参照）。

### hook 実装

```ts
import { useStore } from 'zustand';
import { STATUS_STORE } from '@core/infra/actionProxy';

export function useActionStatus<TValue>(
  action: ActionProxy<TValue>,
): ActionStatus {
  const store = (action as any)[STATUS_STORE];
  return useStore(store);
}
```

`STATUS_STORE` は Symbol キーのため、文字列の step id とは衝突しない。
ActionProxy の Proxy handler で `prop === STATUS_STORE` のときに statusStore を返す（ActionProxy 節参照）。

### runner からの更新タイミング

パイプライン実行フロー（本章）の `runPipeline` が `statusStore.setState(...)` を呼ぶ。

| タイミング | フローステップ | setState の値 |
|---|---|---|
| 実行開始 | 1 | `{ phase: 'running', error: null }` |
| step 失敗 | 5 | `{ phase: 'done', error }` |
| 全成功 + emit ディスパッチ後 | 7 | `{ phase: 'done', error: null }` |

### 宣言的消費と命令的消費の関係

同一の ActionResult から2つの消費経路が導出される:

- **宣言的**（useActionStatus）: statusStore の変更 → Zustand の subscribe → React 再描画
- **命令的**（`await action(value)`）: `runPipeline` の戻り値を直接受け取る

両者は同一の runner が同一のタイミングで生成するため、情報の食い違いは起きない。

## ctx.call と savepoint

doc28 / doc27 の親子間 scope 関係に対応する。

### 確定した方向性

排他 scope は推移閉包でトップレベル Action 開始時に確定する（S11）。
`ctx.call` のリソース管理の役割は不要になった（推移閉包で子の全 Store が親のバッファに含まれる）。
`ctx.call` の役割は savepoint のみである（S12）。

```
親 Action 開始（排他 scope = 推移閉包: { Store1, Store2, Store3 }）
  バッファ: { Store1: {}, Store2: {}, Store3: {} }
  ├── 親 step: ctx.store(Store2).set({ price: 100 })
  │   → バッファ: { Store2: { price: 100 } }
  ├── ctx.call(子)  ← savepoint 取得
  │   ├── 子 step: ctx.store(Store2).set({ price: 200 })
  │   │   → バッファ: { Store2: { price: 200 } }
  │   ├── 子 成功 → savepoint 破棄（書き込みはそのまま）
  │   │   → バッファ: { Store2: { price: 200 } }
  │   └── 子 失敗 → savepoint に復元
  │       → バッファ: { Store2: { price: 100 } }（call 前の状態）
  └── 親 成功 → バッファを Zustand に commit
```

Zustand の state はパイプライン全体を通じて不変であり、
最終的な commit まで一切変わらない。

#### PipelineContext

パイプラインの実行状態を保持する内部構造。
推移閉包により、親子間で同一のバッファを共有する。

```ts
interface EmitEntry {
  signalId: string;
  payload: unknown;
}

/** savepoint: バッファ全体のスナップショット */
interface Savepoint {
  bufferSnapshots: Map<ResourceStore<any>, object>;
  emitQueueLength: number;
}

interface PipelineContext {
  /** 排他 scope（推移閉包）全体のバッファ */
  storeHandles: Map<ResourceStore<any>, BufferedStoreHandle<any>>;
  /** ctx.call の savepoint スタック */
  savepointStack: Savepoint[];
  /** emit() で登録されたシグナルのキュー（commit 後にディスパッチ） */
  emitQueue: EmitEntry[];
}
```

`savepointStack` は初期状態では空配列、`emitQueue` は空配列である。
`ctx.call` で savepoint がスタックに push され、成功時に pop（破棄）、失敗時に pop して復元する。
`emitQueue` へは `ctx.emit()` 呼び出し時にもエントリが追加される。

#### ctx.call の実装

`ctx.call(childAction, value)` は `executeChildCall` で実行する。

**takeSavepoint（ヘルパー）**

現在のバッファ状態をスナップショットとして保存する。

```ts
function takeSavepoint(pipeline: PipelineContext): Savepoint {
  const bufferSnapshots = new Map<ResourceStore<any>, object>();
  for (const [store, handle] of pipeline.storeHandles) {
    bufferSnapshots.set(store, structuredClone(handle.getBuffer()));
  }
  return {
    bufferSnapshots,
    emitQueueLength: pipeline.emitQueue.length,
  };
}
```

※ savepoint の作成には `structuredClone` を使用するため、Store の state には関数・Symbol・DOM ノード等のコピー不能なデータを含めないこと（シリアライズ可能なデータに限定する）。

**executeChildCall**

savepoint の取得・子 ctx の構築・steps 実行・成功/失敗処理を一貫して行う関数。
calls 違反検出もこの関数が担う（runPipeline の ctx.call と子 ctx の再帰的 call の両方から呼ばれるため）。

```ts
// framework/executeChildCall.ts

async function executeChildCall<C>(
  callerDefinition: ActionDefinition<any>,
  pipeline: PipelineContext,
  childAction: ActionProxy<C>,
  childValue: C,
): Promise<void> {
  // --- calls 違反検出 ---
  const resolvedCalls = callerDefinition.scope.calls.map(
    (ref) => (ref as any)[DEFINITION] !== undefined
      ? ref as ActionProxy<any>
      : (ref as () => ActionProxy<any>)()
  );
  if (!resolvedCalls.includes(childAction)) {
    throw new Error(
      `Action "${callerDefinition.id}": ctx.call() の対象 "${childAction.id}" は ` +
        `scope.calls に宣言されていません`
    );
  }

  // --- 子の定義を取得（DEFINITION Symbol 経由。ActionProxy 節参照） ---
  const childDefinition = (childAction as any)[DEFINITION] as ActionDefinition<C>;

  // disabled な子は何もせず正常終了（savepoint 不要）
  if (childDefinition.disabled) return;

  // --- 1. savepoint ��取得 ---
  const savepoint = takeSavepoint(pipeline);
  pipeline.savepointStack.push(savepoint);

  // --- 2. 子の ctx を構築 ---
  //   子は親と同じ storeHandles を使う（推移閉包により全 Store が含まれている）。
  //   子の ctx.store() は、子の internalStores に宣言された Store のみアクセスを許可する。
  const childInternalSet = new Set(flattenRefs(childDefinition.scope.internalStores));
  const childRuntimeMap = new RuntimeMap();

  const childCtx: ActionContext<C> = {
    value: childValue,

    store<T extends object>(ref: ResourceStore<T>): StoreHandle<T> {
      if (!childInternalSet.has(ref)) {
        throw new Error(
          `Action "${childDefinition.id}": ctx.store() に渡された "${ref.label}" は ` +
            `scope.internalStores に宣言されていません`
        );
      }
      return pipeline.storeHandles.get(ref)! as StoreHandle<T>;
    },

    emit(signalId: string, payload?: unknown): void {
      pipeline.emitQueue.push({ signalId, payload });
    },

    async call<CC>(grandchild: ActionProxy<CC>, v: CC): Promise<void> {
      // 再帰的に savepoint を取る（同じ pipeline を共有）
      await executeChildCall(childDefinition, pipeline, grandchild, v);
    },

    getRuntime: (key: string) => childRuntimeMap.get(key),
    setRuntime: (key: string, v: unknown) => childRuntimeMap.set(key, v),
    updateRuntime: (key: string, updater: (current: unknown) => unknown) =>
      childRuntimeMap.update(key, updater),

    // bindContext の解決結果をスプレッド
    ...childDefinition.bindings.resolve(),
  };

  // --- 3. 子の steps を実行 ---
  const composedSteps = compose(childDefinition.steps.toArray());

  try {
    await composedSteps(childCtx);
  } catch (error) {
    // --- 5. 失敗: savepoint に復元して re-throw ---
    const sp = pipeline.savepointStack.pop()!;
    for (const [store, bufSnapshot] of sp.bufferSnapshots) {
      pipeline.storeHandles.get(store)!.restoreBuffer(bufSnapshot);
    }
    pipeline.emitQueue.length = sp.emitQueueLength;
    throw error; // 親が catch すれば続行可能、catch しなければ上位に伝播
  }

  // --- 4. 成功: savepoint を破棄（バッファの書き込みはそのまま残る） ---
  pipeline.savepointStack.pop();
}
```

設計判断:

- 子は親の `pipeline`（storeHandles / savepointStack / emitQueue）をそのまま共有する。推移閉包により親のバッファに全 Store が含まれているため、子専用のバッファは不要
- 子の `ctx.store()` は子の `internalStores` でアクセス制限される（推移閉包のバッファ全体には触れない）
- 子は独自の `RuntimeMap` を持つ（親の runtime とは共有しない）
- 子の `bindContext` は子の定義から解決する（親の bindContext は引き継がない）
- 失敗時は `restoreBuffer(snapshot)` で全バッファを call 前の状態に戻す。`restoreBuffer` は BufferedStoreHandle の Framework 内部専用メソッド
- 失敗時は emit も巻き戻す（`emitQueue.length = sp.emitQueueLength`）
- re-throw により、親の step 関数が try-catch で捕捉するか、そのまま伝播するかを親側が選択できる
- disabled な子を `ctx.call` した場合、savepoint を取らず即座に return する（no-op）

#### ctx.call のフロー図

```
ctx.call(childAction, value):

  排他 scope（推移閉包）: { Store1, Store2, Store3 }（全て親のバッファに含まれる）

  1. savepoint 取得（全バッファの現在値をスナップショット）

  2. 子の ctx 構築（同じ storeHandles を共有、アクセス制御は子の scope で制限）

  3. 子の steps 実行

  4. 成功:
     savepoint 破棄（バッファの書き込みはそのまま）

  5. 失敗:
     savepoint に復元（全バッファを call 前の状態に戻す）
```

#### トップレベル Action の commit フロー

推移閉包により全 Store が単一のバッファセットに含まれるため、
commit フローは簡素化される。

```
6.  全 step 成功 → storeHandles の全ハンドルを commit（全 Store を Zustand に反映）→ snapshot 解放
7.  emitQueue をディスパッチ → { ok: true }
```

推移閉包により `childBuffers` は不要になった。
ステップ 6 の `storeHandles.commit()` で全 Store（子・孫の分を含む）が一括で反映される。

### 失敗時の制御フロー

子 Action が throw した場合の振る舞い:

| シナリオ | バッファ | 親の続行 |
|----------|---------|---------|
| 子 throw → 親 catch しない | 全バッファ破棄（親も throw） | 不可 |
| 子 throw → 親 catch する | savepoint に復元（call 前の状態） | 可能 |

- 子の throw 時は **savepoint に復元** する。子の step が途中で throw した場合の中間状態は整合性が保証されないため、call 前の状態に戻す
- 親が catch すれば、親はバッファ（call 前の状態に復元済み）を維持したまま続行できる
- 親も catch しなければ、エラーはさらに上の階層に伝播し、最終的に runner が catch して全バッファを破棄する
- このルールは再帰的に全ネスト階層に適用される（savepointStack による入れ子）

```
親 Action（排他 scope = 推移閉包: { Store1, Store2, Store3 }）
  バッファ: { Store1: { price: 100 } }
  ├── try {
  │     await ctx.call(子)  ← savepoint 取得
  │     → 子 throw → savepoint に復元
  │   } catch (e) {
  │     → バッファ: { Store1: { price: 100 } }（call 前の状態に復元済み）
  │     → 親は e を見て代替処理を実行できる
  │   }
  └── 親 成功 → commit
```

doc28 の「失敗時は呼び出し前のバッファ状態に復元する」がこの仕組みで実現される。
runner が全例外を catch して ActionResult に変換するため、例外がコンポーネントに漏出することはない。

## シグナルディスパッチ（emit / on）

doc28 のシグナル購読（`ctx.emit` → `on`）に対応する。
emit はブロードキャストであり、購読 Action は独立した top-level Action として実行される。
整合性のある呼び出しが必要な場合は `ctx.call` を使う。

### ctx.emit の実装

`ctx.emit(signalId, payload)` は PipelineContext の `emitQueue` にエントリを追加するだけで即座に return する（`void`）。

```ts
// ctx.emit の実装（ActionContext 構築時に注入）
emit(signalId: string, payload?: unknown): void {
  pipeline.emitQueue.push({ signalId, payload });
}
```

step 内で呼ばれた emit は、呼び出し元パイプラインの commit 後にディスパッチされる。
`ctx.call` 内で emit した場合、子の emitQueue は親にバブルアップされ（ctx.call の成功時処理参照）、
トップレベル Action の commit 後にまとめてディスパッチされる。

### dispatchEmitQueue

パイプライン実行フローのステップ 7 で呼ばれる。

```ts
async function dispatchEmitQueue(queue: EmitEntry[]): Promise<void> {
  for (const entry of queue) {
    const subscribers = findSubscribers(entry.signalId);
    for (const subscriber of subscribers) {
      // 独立した top-level Action として実行（ctx.call ではない）
      await subscriber(entry.payload);
    }
  }
}

function findSubscribers(signalId: string): ActionProxy<any>[] {
  const result: ActionProxy<any>[] = [];
  for (const { proxy, definition } of registry.values()) {
    if (definition.disabled) continue;
    if (definition.on.has(signalId)) {
      result.push(proxy);
    }
  }
  return result;
}
```

- 購読 Action は `on` フィールド（`Set<string>`）にシグナル ID を持つ Action
- `disabled` な Action はスキップする
- 各購読 Action は `subscriber(payload)` で呼ばれる — 通常の Action 呼び出しと同じ。payload が `ctx.value` になる
- 購読 Action は独立した top-level Action であり、自身の snapshot・バッファ・commit を持つ
- 購読 Action の `ActionResult` は emitter の結果に影響しない（emitter は既に commit 済み）
- 購読 Action の失敗は、その Action 自身の `useActionStatus` に反映される

### 実行順序

- 同一シグナル ID に対する複数の購読 Action の実行順序は保証しない
- emit は observer パターンであり、購読 Action 間の順序依存は設計として禁止される
- 実装上は `registry` の列挙順で逐次実行されるが、これは実装の都合であり API の保証ではない

### ctx.emit と ctx.call の対比

| | ctx.call | ctx.emit |
|---|---|---|
| 対象 | 明示的に指定した 1 つの Action | signalId を `on` に持つ全 Action |
| トランザクション | ネスト（親バッファ共有） | 独立（自身の snapshot・commit） |
| 実行タイミング | step 内で即実行 | emitter の commit 後 |
| 失敗の影響 | 親に伝播（throw） | emitter に影響しない |
| 戻り値 | `Promise<void>`（throw で失敗通知） | `void`（キューに積むだけ） |
| 用途 | 整合性が必要な親子呼び出し | ブロードキャスト通知 |

## 定義の確定と遅延適用

S8 より、patch はモジュール評価時に適用され、実行開始前に各 Action の scope が確定する。
本稿で定義した全ミュータブル構造（ResourceStore.extend / StoreGroup.add / StepCollection のミューテーション / ActionProxy のミューテーション / BindingRegistry.bind）は、
モジュール評価フェーズでの使用を前提としている。

### freeze を導入しない理由

ミューテーション API を特定の時点で無効にする freeze の仕組みは導入しない。

パイプライン（runPipeline）は実行開始時に定義を capture する：

1. `computeEffectiveScope(proxy)` で排他 scope を `Set<ResourceStore>` として取得し、各 Store の snapshot を取得する（ステップ 2-3）。以降 `definition.scope` を mutation しても、既に取得した Set と snapshot には影響しない
2. `definition.steps.toArray()`（`[...this.items]` — 浅いコピー）を `compose` に渡し、実行関数チェーンを生成する（ステップ 4）。以降 `definition.steps` を mutation しても、既に合成された関数チェーンには影響しない

この capture 以降、定義への mutation は実行中のパイプラインに影響しない。
したがって、ランタイムで freeze を強制しなくても、パイプラインの構造的な安全性は保たれる。

### Customize の遅延ロード

Customize パッチはモジュール評価時に適用される。
遅延ロード（動的 import）された Customize モジュールのパッチも同様に、モジュール評価時に適用される。

パイプラインの capture により：

- 実行中の Action には遅延適用されたパッチは影響しない
- 次回の Action 実行から反映される

これは freeze の有無に関わらず成立する挙動であり、遅延ロードの Customize を制限する必要はない。

## patchComponent の merge 規則

`patchComponent('slotId', 'componentId', patch)` は
`addComponent` 登録時の props と第 3 引数 patch を Framework が merge し、
コンポーネントに渡す（doc26 §3.1 / §4.1）。merge 規則は以下に従う。

### merge の 2 バケット

merge は 2 つのバケットで独立に行う。

| バケット | 対象 | merge 単位 |
|---|---|---|
| top-level props | patch の top-level キー（`label` / `validation` / `useMain` / 各パーツ固有 props 等） | キー単位 |
| `behaviors` 配下 | patch の `behaviors` オブジェクト内のキー（`useState` / `useValue` / `filter` / `format` 等） | キー単位 |

バケット境界をまたいだ merge（`behaviors` 配下の値をさらに再帰マージする等）は行わない。

### 値の扱い

- 配列・オブジェクト・関数を区別せず、キーの値は単純置換する
- patch のキーが `undefined` の場合、そのキーは merge から除外する（登録時の値を保持する）

単純置換に統一するのは、配列結合・オブジェクト再帰マージ・関数合成といった型別の分岐を Framework 内部に持ち込まず、merge 規則を単一にするためである。型別の合成は customize 側がコードとして明示的に構築する（「合成（元を活かす拡張）の位置づけ」参照）。

### 未登録 id への挙動

指定の `(slotId, componentId)` が未登録の場合、Framework は呼び出し時に例外を投げる。配置操作（`removeComponent` / `replaceComponent` / `insertBeforeComponent` / `insertAfterComponent`）と同じ方針であり、開発 / 本番で挙動差を設けない。silent にスキップしないのは、Core 更新での id 変更や customize 側の typo を気付かず運用するリスクを避けるため（詳細な根拠は doc25「未登録 id への挙動」参照）。

### 複数 patchComponent の衝突

同一の `(slotId, componentId)` に対して複数の `patchComponent` が呼ばれた場合、後勝ちで merge を重ねる。

```ts
patchComponent('slot', 'id', { label: 'A' });
patchComponent('slot', 'id', { label: 'B' });
// → 最終 props.label は 'B'
```

適用順序は doc21 §3フェーズの実行順序（グローバル変更 → Action 登録 + ローカル変更）と、同一フェーズ内では ES module import 順で決定する。衝突時に実行時エラーや警告は出さない。

### 合成（元を活かす拡張）の位置づけ

Framework 側の merge は単純置換で閉じる。「元の値を活かしつつ拡張する」customize は、customize 側が Core の named export を import して値を構築する。

- **配列 props の場合**: doc26 §3.6 の `coreXxxFor()` + spread / filter パターン
- **`behaviors` のキー（部品 hook / plain 関数）の場合**: Core の named export を import してラップした関数を渡す

具体的な書き方と典型シナリオは doc26.5 §3 を参照。Framework 実装は customize 側で構築された最終値をそのまま置換するのみで、合成ロジックを内部に持たない。

### 実装（mergePatch）

```ts
// framework/componentRegistry.ts
function filterUndefined<T extends object>(obj: T): Partial<T> {
  const result: Partial<T> = {};
  for (const k of Object.keys(obj) as (keyof T)[]) {
    if (obj[k] !== undefined) result[k] = obj[k];
  }
  return result;
}

function mergePatch(registered, patch) {
  const { behaviors: patchBehaviors, ...topLevel } = patch;
  return {
    ...registered,
    ...filterUndefined(topLevel),
    behaviors: {
      ...registered.behaviors,
      ...filterUndefined(patchBehaviors ?? {}),
    },
  };
}
```

複数 `patchComponent` は `registered = mergePatch(registered, patch)` の連鎖で累積する。
