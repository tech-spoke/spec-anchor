## 前提

### アーキテクチャ

- フロント: `Astro + Island(React)`
- 管理画面: `React SPA`
- バックエンド: `NestJS`

### ディレクトリ境界

- `core` — upstream / git フォーク管理対象
- `customize` — 利用者カスタマイズ領域

詳細なディレクトリ構造は `22_ディレクトリ構造.md` を参照。

### カスタマイズ性の支配原則

core は upstream / git フォーク管理対象であり、customize は利用者カスタマイズ領域である。この構造のなかで、カスタマイズの自由度を確保しつつ core を保全するという要求は本質的に矛盾する。

この矛盾を解決する支配原則は、**カスタマイズしたい粒度に沿ったカスタマイズ方法を提供する**ことである。

- フィールドの設定を変えたいだけなら、コンポーネント全体を差し替える必要はない（patchComponent）
- ロジックの一部を変えたいだけなら、Action 全体を差し替える必要はない（step 操作）
- パイプライン構造を変えたいなら、それ専用の手段がある（recompose）
- コンポーネント全体を差し替えたいなら、それも可能（replaceComponent）

粒度の細かい手段ほど core の変更に追従しやすく、粗い手段ほど core との分離が進む。大半のカスタマイズが細かい粒度で収まることで、core 更新とカスタマイズの共存が最大化される。

### 基本思想

支配原則から導かれる具体ルール:

- `core` のファイルを直接編集してカスタマイズしない
- `core` は標準構成を提供する
- `customize` は標準構成に差分を当てる
- replace が必要なケースを最小化する（Core 更新とカスタマイズの分離を最大化）

## State Ownership

### UI の入れ子構造

管理画面 UI は次の入れ子構造を持つ。

- ページ
  - セクション（ブロック）
    - サブセクション
      - フィールド

この構造が state owner と Store 設計の根拠になる。

### 原則: 最小の共通親

- field だけで閉じる state は field が持つ
- section 内で協調する state は section が持つ
- 複数 section をまたぐ state は page が持つ

この原則から、全階層（Page / Block / Component）で Scoped Zustand Store を持つ設計が導かれる。

### 原則: 全ての State は Scoped Store に入れる

③ UI状態（サイドバー開閉、タブ選択等）を除き、業務に関わる全ての State（① 正データ・② 一時計算）は Scoped Store に入れる。Scoped Store はコンポーネントにスコープされるため、useState と再レンダリング範囲は実質同じでありパフォーマンス上の不利はない。Store に入れることで Customize 側からの差し替え可能性が確保される。

#### ②と③の境界

ビジネスデータから導出される表示制御は②一時計算であり、③ UI状態ではない。判定基準は送信データへの影響の有無である。

| 例 | 分類 | 理由 |
|---|---|---|
| 商品タイプに応じたセクション表示/非表示 | ② | 非表示セクションのデータは送信対象外になる |
| 権限に基づくタブ無効化 | ② | 無効化されたタブのデータは編集・送信できない |
| サイドバーの開閉 | ③ | 送信データに影響しない |
| タブの初期選択（ナビゲーション目的） | ③ | 送信データに影響しない |
| セクションの折りたたみ（表示はする） | ③ | データは送信される。表示の畳み方だけの問題 |

②に分類されるものは Store + Action パイプラインで扱い、Customize は振る舞い層の API で介入できる。③に分類されるものはコンポーネント層で扱う（doc26 参照）。

### コンポーネントと Action・Store の所有関係

素の React コンポーネントも state（useState）と振る舞い（イベントハンドラ、useEffect）を所有している。本設計はそれを構造化したもの。

```
Component
  ├── calls: Action   … UI イベントから Action を呼ぶ
  └── reads: Store    … Store から状態を読む
```

- **Component → Action**: コンポーネントが Action を呼び出す。UI イベント駆動の基本経路
- **Component → Store**: コンポーネントが Store から状態を読み取り、描画に反映する
- Action は `actionRegistry.register` でモジュールレベルに定義され、名前付き export される。Store はデータの箱であり、Action の scope が参照する資源である

### 原則: 最小の観測主体

state owner と対になる原則。何かの変化を観測して反応する処理は、最小の十分なスコープに置く。

- field の変化だけを見るなら field の Action パイプライン
- section 内の相互依存を見るなら section が子（field）の状態変更を購読し、協調処理の Action パイプラインを動かす
- 送信や受信全体を見るなら page が子（section / field）からの emit を購読し、送信処理の Action パイプラインを動かす

| 階層 | 主な state owner | 主な観測スコープ | 主な責務 |
|------|------------------|------------------|----------|
| field | field | field Action | 入力処理、emit（子→親） |
| section | section | 子（field）の状態変更を購読 | 協調処理、emit（子→親） |
| page | page | 子（section / field）からの emit を購読 | 送信処理 |

### コンポーネントのライフサイクル

全階層（field / section / page）に共通して、コンポーネント自体のライフサイクルは React の mount / update / unmount である。階層による違いはない。

### 各階層の責務（状態・Action・表示・emit / 購読）

#### field

- 状態: フォーカス、ローカルエラー、入力中の整形値、自コンポーネントで閉じる UI 状態
- Action: 入力処理（テキスト、数値、選択、トグル、日付、リッチテキスト、タグ等。API を伴うものを含む — ファイルアップロード、オートコンプリート検索等）
- 表示: 入力コンポーネントの描画、バリデーションエラーの表示
- emit: 親（section または page）へ（入力変更の通知、保存・削除等のボタンクリック、折りたたみトグル等）

#### section

- 状態: 折りたたみ / 展開、子の表示 / 非表示
- Action: 協調処理（表示切替、選択肢の絞り込み等）
- 表示: グループコンテナの描画、子コンポーネントの配置
- 購読: 子（field）の状態変更
- emit: 親（page）へ

#### page

- 状態: submit 状態（loading / disabled）、request / response 状態、redirect 判定
- Action: 送信処理（データ収集・API 発行・結果反映。保存、削除、複製、ステータス変更等）
- 表示: ページレイアウトの描画、セクションの配置、通知（toast 等）
- 購読: 子（section / field）からの emit

## カスタマイズの構造原則

customize 側のコードは、core のディレクトリ構造をミラーした配置を取る。
カスタマイズの方式は、ファイルのサフィックスで宣言する。

| サフィックス | 方式 | 意味 |
|---|---|---|
| `.patch.ts` | patch | core を import し、API で差分を当てる |
| `.ts` / `.tsx`（サフィックスなし） | shadow | core の同名ファイルを丸ごと上書きする（copy & own） |

```
customize/
  store/
    useProductStore.patch.ts       ← Store の拡張（patch）
  presets/
    fieldInput.patch.ts            ← preset の変更（patch）
  actions/
    updatePrice.patch.ts           ← Action の step 操作（patch）
    sendApi.ts                     ← Action の丸ごと差し替え（shadow）
  components/
    products/
      PriceSection.patch.ts        ← コンポーネントの設定変更（patch）
      PriceSection.tsx             ← コンポーネントの丸ごと差し替え（shadow）
```

### 原則

- **置き場所は core のミラー**: customize 側のディレクトリ構造は core と 1:1 に対応する。「どこに置くか」は「何を変えるか」で一意に決まる
- **サフィックスで意図を宣言**: 同じ対象に対して patch と shadow のどちらを使うかは利用者が選ぶ。粒度の判断を利用者に委ねる
- **patch と shadow は同じ対象に対して選択可能**: 少しだけ変えたければ `.patch.ts`、丸ごと書き直したければ `.ts` / `.tsx`。方式の切り替えはファイル名の変更だけで済む

## カスタマイズ手段の全体像

core が提供する標準構成を、customize 側がどう書き換えるか。

| やりたいこと | 手段 | 詳細 |
|---|---|---|
| コンポーネントの追加・削除・並べ替え・入れ替え | 操作 API（add / insertBefore·After / remove / replace） | `25_コンポーネント層（配置操作）.md` |
| コンポーネントの設定変更 | patchComponent（props / useMain / behaviors） | `26.5_コンポーネント層（Customize側）.md` |
| 振る舞いの変更 | Action の id 直接アクセス（`.logic(fn)` / `.wrap(fn)` / `.remove()` 等） | `30_振る舞い層（Customize側）.md` |
| フィールド間の連動 | Action 間シグナル通信（emit / on） | `28_振る舞い層（Core側）.md` |

1つのコンポーネントに対して複数の手段を組み合わせて適用できる。patchComponent で渡す validation は RHF の form 検証経路に接続される（詳細は doc26）。

振る舞いの変更は、Action が持つ id でステップに直接アクセスして操作する。Proxy により `updatePrice.logic(fn)` のようにメソッドとして叩ける。

### カスタマイズの段階

```
軽い                                                                          重い
  ↓                                                                             ↓
  patchComponent → ステップ操作 → Action 全体差し替え → replaceComponent → copy & own
  (props / useMain / behaviors)  (.logic / .wrap / .remove)    (.replace)
```

patchComponent は props の部分上書きと、合成 hook（`useMain`）・内部部品（`behaviors`）の差し替えを同じ API で扱う。大半のカスタマイズは patchComponent とステップ操作で収まる。

copy & own（shadow によるファイル丸ごと差し替え）は最終手段として残るが、patchComponent と replaceComponent でほぼ全てのケースに対応できるため、推奨しない。
