# `/spec-realign` — 仕様編集ワークフロー

ユーザーが提示した課題に対して仕様変更を行う前に、関連する上位概念（プロジェクト目的・アーキテクチャ方針）と関連章を励起してから解決案を構築するワークフロー。**順序を飛ばすと「自分の解決案を正当化する方向にアテンションが流れる」KV バイアスが起きるので、9 ステップを必ず守ること**。

---

## 用語の定義（このコマンド内で一貫して使う）

| 用語 | 内容 | 保持場所 |
|---|---|---|
| **Purpose** | プロジェクトの本来の目的・ビジネスゴール（人が手書き、不変） | `.spec-grag/config.toml` の `core.purpose_dir` で指定されたパス |
| **Concept** | プロジェクトの不変なアーキテクチャ方針・設計思想（spec-grag が更新） | `.spec-grag/config.toml` の `core.concept_dir` で指定されたパス |
| **課題プロンプト** | ユーザーが今回提示した解決すべき問い | このコマンドの引数 `$ARGUMENTS` |
| **章ファイル** | 部分資料・現行仕様の各章（実装レベルの詳細） | `.spec-grag/config.toml` の `sources.include` で指定されたパスパターン（例：`docs/spec/**/*.md`） |
| **章別キーアンカー** | graphrag-rs が章ごとに事前抽出した代表キーワード + 要約。励起クエリで取得する | `.spec-grag/config.toml` の `graph.storage` で指定されたディレクトリ |

**Purpose / Concept の実パスは `spec-grag inject`（② で実行）の出力に含まれるので、それを読み取って参照すること。** 本コマンドではパスを決め打ちしない。

---

## ① 課題プロンプト受領

ユーザーから受領した今回の課題：

> $ARGUMENTS

このプロンプトを「課題プロンプト」として、以降のフローで一貫して参照する。

---

## ② コア同期 + 注入

章ファイルの最新状態を反映し、**Purpose と Concept** を LLM コンテキスト（KV）に再充填する。

!`spec-grag inject`

注：上記出力に **Concept の差分（unified diff）** が含まれる場合は、ここでユーザーに hunk 単位の accept/reject を確認してから ③ に進む。差分なしならそのまま ③ へ。

---

## ③ 章ファイル群の励起（関連章のキーアンカー取得）

### 3-1. Agentic サーチ（章ファイルの直読）

課題プロンプト ($ARGUMENTS) に関連しそうな情報を、`Grep` / `Read` / `Glob` ツールで **章ファイルから直接** 収集する。章ファイルの実際のパスパターンは `.spec-grag/config.toml` の `sources.include` を `Read` して取得すること（プロジェクトごとに異なるので決め打ちしない）。

- 課題プロンプトに登場する識別子・固有名詞でリポジトリ全体を `Grep` する
- ヒットした章ファイルを `Read` して周辺コンテキストを把握する
- 関連しそうな章名・節名を抽出する

### 3-2. キーワードの 2 階層分離抽出（LightRAG Dual-Level Retrieval）

課題プロンプト + 3-1 で収集した情報から、以下の 2 階層に分離してキーワードを抽出する：

- **high_level（抽象概念）**：仕様の議論で使う上位概念。3〜5 個
  - 例：「認証フロー」「データ整合性」「永続化方針」「エラー伝搬」
- **low_level（具体エンティティ）**：コード上の識別子・固有名詞・型名・ファイルパス。5〜10 個
  - 例：「OAuth」「`UserRepository`」「`auth.rs`」「`session_id` カラム」

**抽出基準**：課題プロンプトの解決に「絶対に揺らしてはいけない概念」を high に、「具体的に手を入れる対象」を low に振り分ける。

### 3-3. 励起実行（章別キーアンカー取得）

抽出したキーワードを spec-grag CLI に渡して章別キーアンカーを取得する。**Bash ツールで以下を実行**：

```bash
spec-grag realign "<課題プロンプトを再掲>" \
  --high "high1,high2,high3" \
  --low  "low1,low2,low3,low4,low5"
```

注：**章ファイルの本文そのものは KV に流入させない**。励起出力（章別の keyword + summary 行）のみアテンションに残し、必要な章があれば 3-1 で既に Read した内容を引き継ぐ。

---

## ④ 守るべき制約の導出

課題プロンプト + Purpose + Concept + ③ の章別キーアンカー から、**今回の変更で守るべき制約** を箇条書きで明示する。

各制約は以下の形式：

```
- 制約 N: （一文で記述）
  出典: Purpose / Concept / 章 X のキーアンカー
```

最低 3 個、最大 10 個を目安に。冗長な制約は統合する。

**この書き起こしが KV に制約をアンカーする**（順序の根拠）。書かずに次に進むと、解決案を正当化する方向にアテンションが流れて制約遵守が崩れる。

---

## ⑤ 解決案の構想

課題プロンプト + ④ の制約 → **解決案**。複数案ありえる。各案について：

- 概要（1〜2 文）
- 主要な変更点（箇条書き）
- ④ の制約とどう整合するか

複数案を出した場合は、評価軸（複雑度・整合性・実装コスト）で比較し、**この時点では決定しない**（⑧ の監査を経てから決める）。

---

## ⑥ 波及先の導出（解決案 → 他章への影響範囲）

⑤ の解決案を入力に、再度励起を呼んで **波及先**（解決案を実施した時に影響を受ける他章）を取得する。これは「水平励起」（他章のデータモデル・インターフェース定義との競合検査）にあたる。

設計上、⑥ は「メイン LLM の KV を直接汚染しない」ことが要件。**`Task` ツールで `general-purpose` subagent を起動して以下を投げる**：

```
あなたは spec-grag の波及検査担当です。以下を実行してください：

1. 私（メイン LLM）が出した解決案：<⑤ の解決案サマリを 5〜10 行で記述>
2. この解決案から、章ファイル群の中で影響を受けそうな箇所を励起する。
   high_level / low_level に分けて再抽出し、Bash で以下を実行：
     spec-grag realign "<解決案サマリ>" --high "..." --low "..."
3. 出力された章別キーアンカーから、波及が予想される章とその根拠を 3〜10 件にまとめて返す。

出力フォーマット：
- 章名
- 根拠（どのキーアンカーがマッチしたか、なぜ波及が予想されるか）
```

メイン LLM は subagent の **要約結果のみ** を読む（生の励起出力で KV を汚染しない）。

> 注：subagent 起動コストがあるので、課題が小規模（1 ファイルの修正程度）で波及がほぼないと判断できる場合は省略可。判断はユーザーと相談。

---

## ⑦ 修正解決案の策定

⑤ の解決案 + ⑥ の波及対応 を統合した **修正解決案** を書く。波及先の各章にどう手を入れるか、または「波及先 X は手を入れない、なぜなら～」を明示する。

---

## ⑧ 守るべき制約のチェック（監査役 subagent）

`Task` ツールで **監査役 subagent** を起動し、⑦ の修正解決案を独立 KV で検証する。

**入力**（subagent への指示プロンプトに含めるもの）：
- Purpose 全文（②で取得済み）
- Concept 全文（②で取得済み）
- ④ の守るべき制約（一覧）
- ⑦ の修正解決案

**検証観点**：
1. **垂直整合性**：修正解決案が Purpose / Concept から逸脱していないか
2. **波及検査（水平整合性）**：⑥ の波及先が Purpose / Concept を破壊していないか

**出力**：違反項目 + 総合判定（適合 / 修正必要）。違反があれば違反 ID と是正方向を明示。

**違反があれば ⑤ に戻る。** 反復は最大 3 回までを目安に。3 回で収束しない場合はユーザーに介入を求める。

> 注：専用の監査役 subagent はまだ未実装。**暫定的に `Task` で `subagent_type="general-purpose"` を起動し、上記の検証観点を明示的に指示**して代替する。専用 subagent の実装後にこの記述を書き換える。

---

## ⑨ 最終案の提示

⑧ で適合判定が出たら、最終案をユーザーに提示する。形式：

- 修正解決案の要約（3〜5 文）
- 影響を受ける章とその箇所（一覧）
- 適用手順（diff or 編集箇所のリスト）
- 想定残タスク（人手レビューが必要な項目）

ユーザーが accept したら、**ここで初めて実ファイルへの編集を行う**。スラッシュコマンドの中では編集しない（最終案の合意を取ってから別アクションに移る）。
