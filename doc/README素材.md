# SPEC-anchor 外部仕様書

## 1. SPEC-anchor とは

SPEC-anchor は、LLM (Claude Code / Codex) を使った開発作業で、LLM が本来の目的や仕様から外れることを防ぐツールである。

LLM は、目の前のファイルや直近の会話に強く引っ張られる性質を持つ。背景にある目的や設計原則を見失ったまま作業が進むと、本来の意図と矛盾する回答や修正が出てしまう。SPEC-anchor は、LLM が課題に取りかかる前に、次のものを LLM へ思い出させる。

- プロジェクト本来の目的
- 守るべき設計原則
- 今回の課題に関係する仕様本文
- 章単位の要点
- セクションをまたいで一緒に見るべき関連セクション

利用者は、課題を含む会話文脈がある状態で、LLM が回答や実装を始める前に、SPEC-anchor のコマンドを実行する。SPEC-anchor は保持している情報の中から今回の課題に関係する制約を抽出し、LLM の作業前提として提示する。

### 想定する利用者

- LLM (Claude Code / Codex) を使って実装・調査・設計レビューを行う開発者
- 仕様を管理するプロジェクトオーナー
- 仕様レビュー担当者

### 想定しない利用者

- LLM を使わずに人間だけで仕様を読み書きする運用 (脱線防止が主目的のため、メリットが小さい)

## 2. 利用者ができること

利用者ができることは次の 5 つである。

1. **初期セットアップ**
   - プロジェクトに SPEC-anchor の設定ファイルと、Claude Code / Codex 用の入口を配置する
   - 利用環境に必要な前提が揃っているかを確認する

2. **仕様情報の更新**
   - 仕様本文を書き換えたあと、セクションごとの要約・検索キー・関連セクション・章単位の要点を最新に更新する

3. **作業前の制約注入**
   - 課題を含む会話文脈がある状態で、LLM が回答や実装を始める前に、今回守るべき制約を SPEC-anchor から抽出して LLM に提示させる

4. **制約に沿った回答・修正案の生成**
   - 制約を踏まえた回答案や修正案を LLM に出させる
   - 制約と矛盾する箇所は LLM に明示的に書かせる

5. **仕様衝突の確認と判断**
   - LLM が自力で解決できない仕様の矛盾を「人間判断待ち」として可視化させる
   - 利用者は判断結果を SPEC-anchor へ返し、後続の作業で根拠として使えるようにする

## 3. 基本的な利用の流れ

### 3.1 初回導入

1. プロジェクトに SPEC-anchor をインストールする
2. `spec-anchor-setup-system` を実行し、利用環境に必要な前提が揃っていることを確認する
3. `spec-anchor-setup-project` を実行し、プロジェクトに設定ファイルと LLM 環境用の入口を配置する
4. Purpose ファイルと Core Concept ファイルの中身を、人間の手で書く
5. 仕様本文 (Source Specs) を所定のディレクトリ (既定では `docs/spec/`) に置く
6. `/spec-core` を実行し、最初の保持物を生成する

### 3.2 仕様を更新したとき

1. 仕様本文、Purpose、Core Concept のいずれかを書き換える
2. `/spec-core` を実行する (変更があった部分を中心に、必要な保持物が更新される。変更がない部分は可能な範囲で再利用される)
3. 仕様本文を継続的に編集する場合は、`spec-anchor-watch` を起動して以後の変更を自動更新する (Purpose と Core Concept の編集は自動更新の対象外。編集後に `/spec-core` を実行して反映する)

### 3.3 LLM に作業を依頼するとき

1. LLM の会話文脈に課題を伝える
2. LLM が回答や実装を始める前に `/spec-inject` を実行する
3. LLM は今回の課題に関係する制約を提示する (この時点では回答は出ない)
4. 利用者は提示された制約を確認する
5. 課題に対する回答や修正案が欲しい場合は `/spec-realign` を実行する
6. LLM は制約に従った回答を返し、制約と矛盾する箇所は明示する

### 3.4 仕様の矛盾が出たとき

1. `/spec-core` の実行中に、LLM が自力で解決できない仕様矛盾が検出されると Conflict Review Item として残る
2. 未判断の矛盾がある間、`/spec-inject` と `/spec-realign` は停止する
3. 利用者は提示された矛盾内容を読み、判断結果 (片方を優先 / 両方を満たす条件分岐 / 矛盾ではない / 仕様を直す / 保留 / 今回限り) を返す
4. 解決された矛盾は、後続の `/spec-inject` と `/spec-realign` で根拠として使われる

## 4. コマンド一覧

利用者が直接実行するコマンドは次の 6 つである。

| コマンド | 種別 | 主な用途 |
|---|---|---|
| `/spec-core` | LLM 環境内のコマンド | 保持物の生成・更新 |
| `/spec-inject` | LLM 環境内のコマンド | 課題に対する制約の抽出 (回答は出さない) |
| `/spec-realign` | LLM 環境内のコマンド | 制約を踏まえた回答・修正案の生成 |
| `spec-anchor-watch` | シェルコマンド | 仕様変更を自動で取り込み続ける |
| `spec-anchor-setup-system` | シェルコマンド | 利用環境の前提確認 |
| `spec-anchor-setup-project` | シェルコマンド | プロジェクトへの初期配置 |

`/spec-core` / `/spec-inject` / `/spec-realign` は、Claude Code と Codex のどちらからでも同じ目的で呼び出せる。入口の配置形式は環境ごとに異なり、`spec-anchor-setup-project` が自動で配置する。

- Claude Code → `<プロジェクト>/.claude/commands/` にコマンドテンプレートが置かれる
- Codex → `<プロジェクト>/.codex/skills/spec-anchor/` にスキルが置かれる

---

### `/spec-core`

- **目的**: 仕様本文を読み取り、セクションごとの要約・検索キー・関連セクション・章単位の要点・矛盾候補を生成・更新する。
- **実行するタイミング**: 初回導入時と、仕様本文・Purpose・Core Concept を更新したあと。Source Specs の変更については、`spec-anchor-watch` を起動していれば自動更新される。Purpose または Core Concept を編集した場合は、利用者が `/spec-core` を実行して反映する。
- **利用者が用意するもの**:
  - Purpose ファイル (人間が書く)
  - Core Concept ファイル (人間が書く)
  - 仕様本文 (Source Specs)
  - 設定ファイル (`spec-anchor-setup-project` が作る)
- **実行後に起きること**:
  - 既定では、変更があった部分を中心に必要な保持物が更新される (変更がない部分は可能な範囲で再利用される)
  - 対象の範囲について、セクションごとの要約・検索キー・関連セクション・章単位の要点が再生成される
  - 関連セクションの探索中に仕様矛盾の疑いが見つかり、LLM が自力で解決できない場合は、Conflict Review Item として残る
- **正常時に見る結果**:
  - 更新されたセクションの一覧
  - 生成・更新された情報の種類
  - 検出された矛盾の件数 (あれば)
  - 警告 (あれば)
- **停止時に見る結果**:
  - 失敗した理由 (Purpose / Core Concept が見つからない、仕様本文が 1 件も見つからない、章単位の要点生成に失敗、検索基盤に接続できない、など)
  - 次に取るべき行動の説明
- **停止時に利用者がすること**:
  - 不足しているファイルを補う
  - 利用環境の前提が崩れていれば `spec-anchor-setup-system` で確認する
  - 章単位の要点生成が失敗したら `/spec-core --all` で再試行する
  - 検索基盤の再構築が必要なら `/spec-core --rebuild` を実行する
- **このコマンドがしないこと**:
  - Purpose と Core Concept を自動で書き換えない (常に人間が書く)
  - 仕様本文を自動で書き換えない
  - 検出した仕様矛盾を LLM 判断だけで「解決済み」に変えない

#### 主なオプション

- `--all`: LLM 由来の中間結果を一度捨て、全セクションを再評価する
- `--rebuild`: `--all` に加えて、検索基盤の内容を作り直す
- `--decision-json '<json>'` / `--decision-file <path>`: 未判断の Conflict Review Item に対する判断結果を渡す

### `/spec-inject`

- **目的**: 利用者の現在の課題に関係する制約だけを SPEC-anchor の保持物から抽出し、LLM の作業前提として提示する。回答や修正案は出さない。
- **実行するタイミング**: 課題を含む会話文脈がある状態で、LLM が回答や実装を始める前。
- **利用者が用意するもの**:
  - 課題の内容 (LLM 環境の会話文として)
  - 最新の保持物 (`/spec-core` を実行済みであること)
- **実行後に起きること**:
  - LLM が課題内容を解釈する
  - SPEC-anchor の保持物のうち、今回の課題に関係するものだけが LLM に渡される
  - LLM は今回守るべき制約と、その根拠 (Purpose / Core Concept / 仕様本文の該当箇所 / 解決済みの Conflict Review Item のいずれか) をセットで提示する
- **正常時に見る結果**:
  - **今回守る制約** とそれぞれの根拠
  - **今回見るべき対象**
  - **関連セクションとして確認したもの**
  - **採用しなかった候補と理由**
  - **不確実性 / 人間確認が必要な点**
- **停止時に見る結果**:
  - 「保持物が古いので更新が必要」「自動更新中なので待機が必要」「未判断の仕様矛盾があるので判断が必要」「保持物の一部生成に失敗している」のいずれかの理由
  - 次に取るべき行動の指示
- **停止時に利用者がすること**:
  - 仕様変更が未反映なら `/spec-core` を実行する
  - 自動更新中なら完了を待つ
  - 未判断の矛盾があれば §7 の手順で判断を返す
  - 保持物が壊れていれば `/spec-core --all` で再生成する
- **このコマンドがしないこと**:
  - 課題への回答や修正案を作らない
  - 検索キーや要約だけを根拠に制約を確定しない (根拠は常に Purpose / Core Concept / 仕様本文 / 解決済みの矛盾のいずれか)

---

### `/spec-realign`

- **目的**: `/spec-inject` と同じ手順で制約を抽出し、その制約に従った回答案または修正案を作る。
- **実行するタイミング**: 課題に対する具体的な回答や修正コードを LLM に出してほしいとき。
- **利用者が用意するもの**:
  - 課題の内容
  - 最新の保持物
  - (推奨) 事前に `/spec-inject` を実行して制約を確認しておく
- **実行後に起きること**:
  - `/spec-inject` と同じ制約抽出が行われる
  - 抽出された制約に従って、LLM が回答案または修正案を作る
  - 制約と矛盾する部分は、「競合 / 不確実性 / 人間レビューが必要な点」として回答内に明示される
- **正常時に見る結果**:
  - **今回守る制約**
  - **今回扱う修正候補または検討対象**
  - **競合 / 不確実性 / 人間レビューが必要な点**
  - **課題への回答または修正案**
- **停止時に見る結果**:
  - `/spec-inject` と同じ停止理由
  - 加えて、LLM が課題の中心を特定できなかった場合は、利用者に追加情報を求めるメッセージが出る
- **停止時に利用者がすること**:
  - `/spec-inject` と同じ対処を行う
  - 課題の中心が特定できない場合は、より具体的に課題を伝え直す
- **このコマンドがしないこと**:
  - 制約と矛盾する回答を、矛盾を隠した形で出さない (必ず明示する)
  - Purpose、Core Concept、仕様本文を自動で書き換えない

### `spec-anchor-watch`

- **目的**: 仕様本文 (Source Specs) の変更を自動で検知し、`/spec-core` 相当の更新をバックグラウンドで実行する。Purpose または Core Concept を編集した場合は、自動更新の対象外であり、利用者が `/spec-core` を実行して反映する。
- **実行するタイミング**: 仕様本文を継続的に書き換える間。明示的に止めるまで動き続ける。
- **利用者が用意するもの**:
  - `spec-anchor-setup-project` を済ませたプロジェクト
  - 対象プロジェクトのディレクトリ (`cd` してから起動する)
- **実行後に起きること**:
  - 指定間隔で仕様本文を監視する
  - 変更を検知すると、まとめて 1 回の更新サイクルとして処理する
  - 更新中は、`/spec-inject` と `/spec-realign` は安全のため停止する
- **正常時に見る結果**:
  - 各更新サイクルの結果が逐次表示される
  - 更新されたセクション、生成された情報、検出された矛盾の件数
- **停止時に見る結果**:
  - 設定ファイルが見つからない、古いロックが残っている、などのエラー
- **停止時に利用者がすること**:
  - 設定ファイルが無ければ `spec-anchor-setup-project` で初期化する
  - 古いロックが原因なら、`--stale-lock-sec` を調整するか、watcher を再起動する
- **このコマンドがしないこと**:
  - LLM への課題対応 (制約注入や回答生成) は行わない
  - Purpose と Core Concept を自動で書き換えない

#### 主なオプション

- `--once`: 1 回だけ更新して終了する
- `--interval-sec <秒>`: 変更がないときの監視間隔
- `--debounce-sec <秒>`: 連続変更を 1 回の更新にまとめる待ち時間
- `--max-runs <回数>`: 指定回数だけ更新したら終了する

### `spec-anchor-setup-system`

- **目的**: 利用環境に SPEC-anchor の動作前提が揃っているかを確認する。
- **実行するタイミング**: 初回導入時、または利用環境が変わったあとの再確認時。
- **利用者が用意するもの**: なし (既存環境を対象に動作する)
- **実行後に起きること**:
  - SPEC-anchor 本体のコマンドが実行可能かを確認する
  - 仕様本文の検索基盤に接続できるかを確認する
  - Claude / Codex の CLI が利用可能かを確認する
  - 利用者のファイル・設定は一切変更しない (確認専用)
- **正常時に見る結果**:
  - 「利用可能 (ready)」と、各前提の確認結果一覧
- **停止時に見る結果**:
  - 「不足あり (blocked)」と、不足している前提の一覧
  - それぞれについて何が不足しているかの内訳
- **停止時に利用者がすること**:
  - 不足している CLI をインストールする
  - 検索基盤のサービスを起動する
  - 必要なライブラリを導入する
- **このコマンドがしないこと**:
  - 設定ファイルや保持物に書き込まない
  - 不足している前提を勝手にインストールしない

#### 主なオプション

- `--check-only`: 確認のみを行う (規定動作。明示する場合に使う)
- `--qdrant-url <url>`: 検索基盤の接続先を明示的に指定する

---

### `spec-anchor-setup-project`

- **目的**: 対象プロジェクトに SPEC-anchor の設定ファイル、Purpose / Core Concept の雛形、Claude Code / Codex 用の入口を配置する。
- **実行するタイミング**: 初回導入時、または別プロジェクトに SPEC-anchor を導入するとき。
- **利用者が用意するもの**:
  - 配置先のプロジェクトディレクトリ (既に存在している必要がある)
- **実行後に起きること**:
  - `.spec-anchor/config.toml` が作られる
  - Purpose と Core Concept の雛形ファイルが作られる
  - Claude Code 用のコマンドテンプレートと Codex 用のスキルが、`--agent` の指定に応じて配置される
  - `.gitignore` の推奨設定が作られる
- **正常時に見る結果**:
  - 作成・更新されたファイルの一覧
  - 既存のため保護されて上書きされなかったファイルの一覧
- **停止時に見る結果**:
  - 配置先が存在しない、またはディレクトリでない、というエラー
  - 既存ファイルと内容が衝突した場合の差分
- **停止時に利用者がすること**:
  - 配置先ディレクトリを先に作る
  - 衝突した既存ファイルを確認し、上書きしてよければ `--force` で再実行する
  - 慎重に進めたい場合は `--dry-run` で変更予定を先に確認する
- **このコマンドがしないこと**:
  - Purpose と Core Concept の中身を書かない (雛形だけ作る。中身は人間が書く)
  - `/spec-core` を自動で実行しない (配置完了後に利用者が明示的に実行する)

#### 主なオプション

- `--target <path>`: 配置先プロジェクトディレクトリ (既定はカレントディレクトリ)
- `--agent <codex|claude|both>`: 配置する LLM 環境の入口 (既定は両方)
- `--dry-run`: 実際には配置せず、変更予定だけを表示する
- `--force`: 既存ファイルがあっても上書きする
- `--no-init-core-files`: Purpose と Core Concept の雛形を作らない

## 5. 保持される情報

SPEC-anchor が保持する情報は次の通り。利用者は「誰が更新するか」「何のために使うか」「制約の根拠として使ってよいか」で区別する。

### 5.1 Purpose (目的)

- **誰が更新するか**: 人間
- **何のために使うか**: プロジェクトの存在理由、達成したい価値、上位の判断基準を示す
- **制約の根拠として使えるか**: はい (最上位の根拠)

SPEC-anchor は Purpose を自動で書き換えない。中身を変えたい場合は、利用者が直接ファイルを編集する。

### 5.2 Core Concept (中核コンセプト)

- **誰が更新するか**: 人間
- **何のために使うか**: プロジェクト全体の判断軸、承認済みの設計原則、不変に近い方針を示す
- **制約の根拠として使えるか**: はい

SPEC-anchor は Core Concept を自動で書き換えない。仕様が変わっても、Core Concept が古くなったかどうかの自動通知は出さない。更新の判断は常に人間が行う。

### 5.3 Source Specs (仕様本文)

- **誰が更新するか**: 人間
- **何のために使うか**: 現在の仕様そのもの。LLM が課題を解く際の「守るべき既存仕様」または「修正対象」になる
- **制約の根拠として使えるか**: はい

仕様本文は Markdown 見出しを境界として、セクション単位に分割される。分割の最大深さは設定で変更できる (既定では 4 段階目まで)。それより深い見出しは独立したセクションにはならず、直近の親セクションに取り込まれる。

本書では、仕様本文を Markdown 見出しで区切った単位を「セクション」と呼ぶ。「章単位の要点」だけは、複数のセクションをまとめた章 (最上位見出しの範囲) 単位の情報として扱う。

### 5.4 セクションごとの要約

- **誰が更新するか**: `/spec-core` が自動生成・更新
- **何のために使うか**: 各セクションが何について書かれているかを LLM が短時間で把握する
- **制約の根拠として使えるか**: いいえ (検索と理解の補助)

制約の根拠にしたい場合は、要約ではなく、対応する仕様本文の該当箇所を確認する。

### 5.5 検索キー

- **誰が更新するか**: `/spec-core` が自動生成・更新
- **何のために使うか**: 課題に関係しそうなセクションを見つけ出すための検索語
- **制約の根拠として使えるか**: いいえ (検索のための補助語)

### 5.6 関連セクション

- **誰が更新するか**: `/spec-core` が自動生成・更新
- **何のために使うか**: あるセクションを見るときに、依存・影響・同じ方針・衝突の疑いなどの理由で一緒に見るべきセクションの一覧を示す
- **制約の根拠として使えるか**: 関連セクションそのものは根拠ではない。関連セクションをたどって参照した仕様本文の該当箇所は根拠になる

LLM は関連セクションを再帰的にたどれる。どこまでたどるかは LLM 自身の判断による。SPEC-anchor は無制限のたどり込みをしない。

### 5.7 章単位の要点

- **誰が更新するか**: `/spec-core` が自動生成・更新
- **何のために使うか**: 章全体の重要テーマ、判断軸、章内で見るべき主要セクションへの入口を示す
- **制約の根拠として使えるか**: いいえ (入口情報)

要点で関係しそうな章を特定したあとは、対応する仕様本文を確認して根拠を取る。

### 5.8 Conflict Review Item (仕様矛盾の判断待ち)

- **誰が更新するか**: `/spec-core` が候補を生成し、最終判断は人間が返す
- **何のために使うか**: LLM が自力で解決できない仕様矛盾を、人間判断待ちとして可視化する
- **制約の根拠として使えるか**: 「解決済み」かつ古くなっていないものは根拠として使える。「未判断」のものは根拠にならず、後続の `/spec-inject` と `/spec-realign` を停止させる

詳細は §7 を参照。

## 6. 鮮度確認と停止条件

SPEC-anchor は、保持物が最新でない、または安全に作業を進められない状態のとき、`/spec-inject` と `/spec-realign` を停止させる。利用者は停止理由を確認し、対処してから再実行する。

### 6.1 停止条件と対処

| 状態 | `/spec-inject` と `/spec-realign` の振る舞い | 利用者の対処 |
|---|---|---|
| 仕様本文を変えたが保持物に反映されていない | 停止する | `/spec-core` を実行する |
| `spec-anchor-watch` が動作中、または未処理の変更が残っている | 停止する | 自動更新の完了を待つ |
| 未判断の Conflict Review Item がある | 停止する | §7 の手順で判断を返す |
| 一部の保持物が生成に失敗している | 停止する | `/spec-core --all` で再生成する |
| 任意の保持物の一部が欠けているが必須分は使える | 警告を出して続行する | 必要なら `/spec-core` で補完する |

Purpose または Core Concept を編集した場合は、自動更新の対象外である。利用者は `/spec-core` を実行してから `/spec-inject` と `/spec-realign` を使う。

### 6.2 仕様変更と未判断の矛盾が同時にある場合

両方ある場合は、先に `/spec-core` を実行する。仕様変更を反映する過程で矛盾が自動的に解消されることもあるため、人間判断が必要な矛盾は更新後に残ったものに限定される。

### 6.3 自動で行われないこと

- `/spec-inject` と `/spec-realign` は、停止状態でも `/spec-core` を自動で起動しない。保持物の更新は、常に利用者または自動更新プロセスのいずれかが制御する。
- 未判断の Conflict Review Item を LLM 判断だけで「解決済み」に変えない。

## 7. Conflict Review Item

### 7.1 どういうものか

`/spec-core` の実行中に、Purpose、Core Concept、Source Specs の根拠同士が同時に満たせない疑いを LLM が検出し、かつ既存の根拠だけでは解決できないと判断した場合に、Conflict Review Item として記録される。

### 7.2 利用者に提示される内容

各 Conflict Review Item には次が含まれる。

- 矛盾している箇所の出典 (どの仕様ファイルのどのセクションか)
- それぞれの主張の要約
- 矛盾していると判断した理由
- LLM が自力で解決できない理由
- 関連するセクション
- 利用者が選べる判断肢
- 次に取るべき行動の推奨

### 7.3 利用者が返す判断肢

利用者は次のいずれかを返す。

| 判断肢 | 意味 | 結果 |
|---|---|---|
| 片方の仕様を優先する | 矛盾するうちの 1 つを正とする | 解決済み |
| 両方を満たす条件分岐を指示する | 条件によって使い分ける | 解決済み |
| 矛盾ではないとして取り下げる | 誤検知だった | 取り下げ |
| 仕様を直す必要があると差し戻す | 仕様本文 / Purpose / Core Concept の修正が必要 | 未判断のまま (修正後に再評価) |
| 今回は判断保留にする | 後で判断する | 未判断のまま |
| 今回の課題内だけの一時判断にする | この課題に限って判断、後続セッションへ持ち越さない | 解決済み (適用範囲は今回限り) |

### 7.4 判断結果の反映

- 「解決済み」になった Conflict Review Item は、後続の `/spec-inject` と `/spec-realign` で根拠として使える
- 判断のあとに対象の仕様本文 / Purpose / Core Concept が書き換わると、その判断は古くなったものとして根拠から外される
- SPEC-anchor は判断結果を仕様本文に自動で反映しない (反映は人間が直接編集する)

### 7.5 未判断のまま残すとどうなるか

- 未判断の Conflict Review Item が残っている間、`/spec-inject` と `/spec-realign` は停止する
- 「無視して進む」「LLM 判断で進める」などの自動迂回は行われない (人間判断が必要)

## 8. 保証すること / 保証しないこと

### 8.1 SPEC-anchor が保証すること

- Purpose と Core Concept は人間管理対象であり、自動で書き換えない
- 仕様本文を自動で書き換えない
- 保持物が古い、または一部失敗している状態で `/spec-inject` と `/spec-realign` を進めない
- 未判断の Conflict Review Item を残したまま `/spec-inject` と `/spec-realign` を進めない
- 制約を提示するとき、Purpose / Core Concept / 仕様本文 / 解決済みの Conflict Review Item のいずれかの根拠を必ず示す
- 制約と矛盾する回答案を、矛盾を隠した形では出さない (必ず明示する)
- 同じ仕様情報に対して、Claude Code と Codex は同じ保持物とコマンド契約を利用できる (ただし、LLM ごとの解釈や回答内容の完全一致は保証しない)

### 8.2 SPEC-anchor が保証しないこと

- LLM が提示する制約と回答案が常に正しいこと (利用者の確認が必要)
- 仕様変更後に Purpose と Core Concept が陳腐化していないかの自動検出 (人間判断による)
- 自動検出される仕様矛盾の網羅性 (検出されない矛盾が残る可能性がある)
- 検索キーと要約だけで関係するセクションをすべて見つけ出すこと (補助情報であり、根拠ではない)

### 8.3 利用者が必ず確認するべきこと

- `/spec-inject` が提示した制約が、今回の課題に対して妥当か
- `/spec-realign` の回答案が、Purpose / Core Concept / 仕様本文と矛盾していないか
- Conflict Review Item で返した判断が、その後の仕様変更で古くなっていないか

## 9. エラー時の見え方と対処

SPEC-anchor が停止すると、画面には「何が起きたか」と「次に何をすればよいか」が表示される。利用者は表示に従って対処する。

### 9.1 設定ファイルが見つからない

- 見え方: 「`.spec-anchor/config.toml` が見つかりません」というメッセージ
- 対処: `spec-anchor-setup-project --target <プロジェクトのパス>` を実行して初期化する

### 9.2 Purpose または Core Concept のファイルが見つからない

- 見え方: 「指定された Purpose / Core Concept のファイルが見つかりません」というメッセージ
- 対処: 該当ファイルを作成し、人間が中身を書く

### 9.3 仕様本文が 1 件も見つからない

- 見え方: 「対象となる仕様本文が見つかりません」というメッセージ
- 対処: 仕様本文を所定のディレクトリに置く。または設定の `[sources].include` を実在のパスに合わせる

### 9.4 章単位の要点の生成に失敗した

- 見え方: 「章単位の要点の生成に失敗しました。前回の値が保持されています」というメッセージ
- 対処: `/spec-core --all` で再試行する

### 9.5 仕様本文の検索基盤に接続できない

- 見え方: 「検索基盤に接続できません」というメッセージ
- 対処: 検索基盤のサービスを起動する。復旧後に `/spec-core --rebuild` を実行する

### 9.6 検索基盤の内容に不整合がある

- 見え方: 「検索基盤の内容と保持物が一致しません」というメッセージ
- 対処: `/spec-core --rebuild` を実行する

### 9.7 保持物が古い

- 見え方: 「仕様本文が変わっていますが、保持物が更新されていません」というメッセージ
- 対処: `/spec-core` を実行する

### 9.8 自動更新中で待機が必要

- 見え方: 「自動更新が動作中です」というメッセージ
- 対処: 自動更新の完了を待ってから再実行する

### 9.9 未判断の Conflict Review Item がある

- 見え方: 矛盾の内容と判断肢が表示される
- 対処: §7 の手順に従って判断を返す

### 9.10 課題の中心が特定できない (`/spec-realign` のみ)

- 見え方: 「課題の中心が特定できません。より具体的な情報をお願いします」というメッセージ
- 対処: 課題をより具体的に伝え直す。事前に `/spec-inject` で制約を確認しておくと有効

### 9.11 配置先プロジェクトで既存ファイルが衝突する (`spec-anchor-setup-project` のみ)

- 見え方: 衝突したファイルと、提案される変更の差分
- 対処: 内容を確認し、上書きしてよければ `--force` で再実行する。慎重に進めたい場合は `--dry-run` で事前確認する

### 9.12 利用環境の前提が不足している (`spec-anchor-setup-system` のみ)

- 見え方: 不足している前提の一覧 (LLM CLI、検索基盤、必要ライブラリなど)
- 対処: 表示されたものを順に整備する。整備後に再度 `spec-anchor-setup-system` で確認する
