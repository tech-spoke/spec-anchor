# 内部設計書の記述ルール

本文書は、`doc/DESIGN.ja.md` を記述・レビューする際のルールを定める。

## 内部設計の記述ルール

内部設計は実装メモではない。

内部設計は、外部設計を実現するために決定された内部構造を記述しなければならない。

内部設計は、調査、実装作業、デバッグ、試行錯誤、一時的な推論のログではなく、設計判断の結果として記述しなければならない。

内部設計に含めてよいもの:

- モジュールの責務
- コンポーネントの境界
- データモデルとその意味
- 状態遷移
- 処理フロー
- 依存関係の方向
- エラー処理方針
- 永続化と再生成のルール
- 不変条件
- 内部コンポーネント間のインターフェース
- 外部動作と内部責務の対応関係

内部設計に含めてはならないもの:

- 実装日誌
- デバッグメモ
- 一時的な回避策
- 設計根拠として必要な場合を除く、却下された案
- テスト専用の設定
- プロバイダー固有のコマンド詳細
- 設計判断を伴わない TODO
- コーディングエージェントの作業履歴
- 段階的な実装手順
- 「後で処理する」「要整理」「一時的」などの曖昧なメモ

そのような情報が有用な場合は、実装メモ、設計根拠、テストメモ、またはタスクリストへ移動すること。

## 設計判断の結果ルール

内部設計は、確定した設計判断を記録しなければならない。

重要な内部設計の記述ごとに、Agent は次の問いに答えなければならない。

1. 何が決定されたのか？
2. どの外部動作または要件を支えるのか？
3. どの内部コンポーネントが責任を持つのか？
4. どのデータまたは状態を所有するのか？
5. どの入力を受け取るのか？
6. どの出力を生成するのか？
7. どの不変条件が維持されなければならないのか？
8. 失敗時には何が起きるのか？

Agent は、内部設計を思考過程の記録として書いてはならない。

悪い例:

> 既存のグラフ成果物を使うことを検討したが、一部の stale ケースが難しかったため、後で整理が必要かもしれない。

良い例:

> グラフ更新コンポーネントは、ソースセクション由来の古いノードの削除に責任を持つ。
> ソースセクションから作成されるすべてのノードは、`source_section_id` と `extract_run_id` を保持しなければならない。
> 差分更新時には、同じ `source_section_id` を持ち、かつ古い `extract_run_id` を持つノードを、新しい抽出結果を書き込む前に削除する。

## 設計メモから内部設計への変換ルール

設計メモを内部設計へ変換するとき、Agent はメモをそのままコピーしてはならない。

Agent は、確定した判断だけを抽出し、安定した設計記述として書き直さなければならない。

各メモを次のいずれかに分類すること。

- 確定した判断
- 設計根拠
- 却下された選択肢
- 未解決の論点
- 実装メモ
- テストメモ
- タスク項目
- 廃止されたメモ

内部設計の本文に書いてよいのは、確定した判断のみである。

設計根拠は、別の「Rationale」セクションに置いてよい。
未解決の論点は、別の「Open questions」セクションに置かなければならない。
実装詳細は、実装メモに置かなければならない。
タスク項目は、内部設計の本文ではなく、タスクリストに置かなければならない。
