# 外部仕様書リライト指針

## 0. 改訂の経緯

旧版の本指針は「外部仕様書 = 利用者が読みやすい形に書き直したもの」という方向で書かれており、Agent が指針に従って作成した結果、外部契約書ではなく利用者ガイド (Getting Started / README) に近い文書ができあがった。Agent はこの指針を素直に踏襲して、CLI option / stdout JSON field / exit code / artifact 更新条件 / 状態遷移 / error reason code / 設定キーといった契約情報を「内部設計へ移す」「読みづらいので落とす」と判断し、結果として外部から観測できる契約のかなりの部分が本文から消えた。

本指針はこの失敗を踏まえ、外部仕様書を「人間が読める外部契約書」として書くための基準を再定義する。

## 1. 外部仕様書の定義

外部仕様書は、SPEC-anchor が外部から観測できる入出力契約を、人間が読める形で書いた仕様書である。

**含むもの**

- 利用者・運用者・Agent が入力する内容
- 利用者・運用者・Agent が読む出力
- 外部から観測できる状態と遷移
- 外部から観測できるエラー条件と表示内容
- 設定ファイル経由で外部から与えられる入力 (全 table / 全 key)
- Agent / 利用者が打鍵する識別子の形式
- 互換性上の保証範囲

**含まないもの**

- 内部実装 (アルゴリズム、保存形式、内部 stage、provider routing 実装)
- 検証管理情報 (どの test で検証するか、検証進捗マーク)
- 方式分類や実装方式の妥当性説明
- 議論の経緯、過去案の経緯、作業メモ

利用者ガイド (README / Getting Started) とは別文書である。外部仕様書を読み物として読みやすくするために契約情報 (CLI option、stdout JSON field、exit code、error reason code、状態遷移、設定キーなど) を削ってはいけない。

「人間向け」は「利用者ガイド化」ではない。「人間が読める外部契約書」である。読みやすさよりも契約の網羅性と正確性を優先する。

## 2. 外部仕様書に必ず残す情報

次の情報は読みやすさのために削除してはならない。すべて契約として外部に観測される。

### 2.1 コマンド契約

- CLI コマンド名
- サポートする全オプション (任意・必須・既定値を含む)
- 入力ファイル / 入力 JSON / decision payload の schema
- stdout JSON の top-level field 名と意味
- stdout JSON / artifact JSON の nested field のうち、Agent / 利用者 / 後続コマンドが参照するもの (例: `freshness_report.blocking_reasons[]`、`conflict_review_items[].status`、`warnings[]`、`recommended_next_action`)
- stderr に出る情報の種類
- CLI exit code
- 利用者に表示される停止理由
- error reason code (機械可読 enum 値)

stdout JSON / artifact JSON は、top-level field だけでなく、Agent / 利用者 / 後続コマンドが参照する nested field も外部仕様として定義する。内部処理専用で、Agent / 利用者 / 後続コマンドのいずれも参照しない field は内部設計に移す。

### 2.2 artifact 契約

外部仕様に残すのは「利用者・Agent・後続コマンドが参照する path、status、field、更新条件」である。

- 作成・更新される artifact path
- 作成条件
- 更新条件
- 更新しない条件 (前回値が保持される条件)
- artifact の status 値とその遷移
- artifact 内に格納される field のうち、Agent / 利用者 / 後続コマンドが参照するもの (top-level / nested 問わず)

内部設計に移すのは「その artifact を生成・永続化する内部データ構造、内部 index、payload、cache、fingerprint、provider 依存の保存方式」である。この境界を守ることで、「artifact field を消す」誤りと「payload 内部まで全部外部仕様に出す」誤りの両方を防ぐ。

### 2.3 設定ファイル契約

`.spec-anchor/config.toml` の全 table / 全 key を外部仕様の対象として扱う。「内部 tuning だから省略」「ふつうは触らないから落とす」という選別はしない。`[sources]` / `[core]` / `[section]` / `[section_metadata]` / `[chapter_anchor]` / `[llm.providers]` / `[llm.stage_routing]` / `[retrieval]` / `[embedding]` / `[vector_store]` / `[limits]` / `[watcher]` のすべてが対象。

各キーについて外部仕様に書く粒度は次に限定する:

- 型
- 必須 / 任意
- 既定値
- 許可値 (enum / 範囲)
- 不正値時のエラー
- 外部から観測できる効果 (設定を変えたとき、何が変わるか)
- 互換性上、変更すると breaking change になるか

逆に、設定値を内部でどう使うか、どの provider をどう呼ぶか、検索や embedding をどう実装するか、payload や cache をどう構成するかは内部設計に移す。

### 2.4 識別子契約

Agent / 利用者が打鍵する識別子 (`source_section_id` など) について、外部仕様に必ず書く項目:

- 形式 (生成規則を含む)
- 一意性の保証範囲 (どの集合内で一意か)
- どのコマンド入力で使うか (`spec-anchor inject-section "<id>"` など)
- どの出力 field に現れるか (`section_id` / `source_section_id` 等の field 名)
- 見つからない場合の reason code とエラー表現
- 入力対象 (heading 変更 / section 並び替え / 章追加など) によって値が変わりうるかの保証範囲

内部の UUID 変換、point id 生成方式、namespace 値、payload 内での扱いなどは内部設計に移す。

### 2.5 freshness gate 契約

- 停止条件 (status 値と blocking_reasons の組み合わせ)
- 各停止理由に対する recommended_next_action
- 後続コマンドが進めるかどうかの判定基準

### 2.6 Conflict Review Item 契約

- status 値 (pending / dismissed など) とその遷移
- decision payload の構造と必須 field
- 各 decision 値の意味と結果
- 却下の失効判定条件
- valid_scope の値と適用範囲

### 2.7 副作用契約 (negative contract)

- 自動で行わないこと
- 上書きしないファイル
- 黙って fallback / 別 provider 切替しないこと
- 自動で「解決済み」「最新」に状態を変えないこと

### 2.8 互換性ラベル

各契約項目 (設定キー / CLI option / JSON field / status / reason / artifact path / 状態遷移) には、変更時の影響範囲を示す互換性ラベルを付ける。原則は次の 4 段階。

- `stable`: 変更すると既存利用者・Agent・テストに影響する外部契約。breaking change として扱う
- `advanced-stable`: 高度な運用設定だが、key / 型 / 既定値 / エラー挙動は外部契約。`stable` と同じく breaking change として扱う
- `experimental`: 利用可能だが互換性保証が弱い。本文にラベルを明記し、利用者に注意喚起する
- `internal-observable`: 内部寄りだが artifact / JSON / 設定として外部から観測される。削除・変更時は影響範囲を本文で明記する

ラベルを付ける目的は 2 つ。利用者が「どの契約に依存してよいか」を判定できるようにすること。SPEC-anchor 側の変更時に breaking change の範囲を見落とさないようにすること。ラベルは項目単位で付ける (テーブル列か注記で明示する)。

## 3. 外部仕様書に書かないもの

次は内部設計書に移す。外部仕様書本文には書かない。

- 内部アルゴリズム (検索方式、ranking、embedding 入力 text の組み立て、point id 生成、cache key 生成、stage 実行順、retry / debounce の内部ロジック)
- 保存の内部構造 (Qdrant payload field の内訳、UUID5 namespace、fingerprint の組み立て方、state file の内部 field のうち Agent / 利用者が読まないもの)
- provider routing の実装方法
- 方式分類や業界用語との対比 (Hybrid RAG / property graph / hierarchical cluster などの実装方式説明)
- performance tuning の理屈
- 内部 stage 名 (原則として内部設計に置く。ただし `core_progress.json` などの外部 artifact に出力され、Agent / 利用者 / 後続コマンドが参照する stage 名は §2.1 / §2.2 に従って外部仕様に残す。stage 名の追加・削除・改名は `internal-observable` の互換性ラベル (§2.8) を付けて影響範囲を明示する)

検証管理情報 (検証進捗マーク `[ ]` / `✅`、どの test file で検証するか、tool call trace 監査ケース) は外部仕様書本文に書かない。検証管理は別途テスト計画書側 (`doc/e2eテスト/test_plan.ja.md` など) で運用する。

## 4. 残す / 落とすの判断基準

判断に迷ったら次の問いを順に当てる。

**外部仕様書に残す** (1 つでも yes なら残す)

1. 利用者・運用者・Agent が入力するものか
2. 利用者・運用者・Agent が読む出力か
3. コマンドの成功・失敗・停止の判定に必要か
4. 後続コマンドが進めるかどうかを決める状態か
5. 互換性を壊す変更になりうる契約か

**内部設計に移す** (1 つでも yes なら内部)

1. その出力を作るための内部アルゴリズムか
2. 保存・検索・生成の内部実装か
3. provider / cache / payload の実装詳細か
4. テストでどう確認するかの話か
5. 方式分類や実装方針の妥当性説明か

両方の判定が yes になる項目がありうる (例: 設定キー)。その場合は「何を外部に書き、何を内部に書くか」を粒度で分ける。外部仕様には「キー、型、必須性、既定値、許可値、不正値時のエラー、観測効果」までを書き、その値を内部でどう使うかは内部設計に書く。

迷ったら外部に残す。落とすときは「外部からは観測できないから内部に書く」と理由を内部設計側に明示する。

### 4.1 情報源の優先順位

外部仕様ドラフトを作成するときは、次の優先順位で根拠を扱う。

1. 現在の実装で外部から観測できる入出力 (CLI を実際に動かして得られる stdout JSON、CLI exit code、artifact ファイル内容)
2. 既存 `doc/EXTERNAL_DESIGN.ja.md` に明記された外部契約
3. テストで固定されている外部挙動 (`tests/` 配下の assertion で固定された値、fixture)
4. `doc/README素材.md` などの利用者ガイド素材、または過去の議論ログ

上位と矛盾する内容は、下位の情報源だけを根拠に断定値として採用しない。実装・既存設計書・テストの間で矛盾がある場合は、本文に断定せず §4.2 の形式で「未確定事項」として分離する。

### 4.2 未確定事項の扱い

外部契約として断定できない項目は推測で埋めない。次のいずれかに該当する場合は、本文の契約表に断定値を書かず、各章末尾の「未確定事項」節として分離する。

- 実装で観測できない (CLI を動かしても確認できない)
- 既存 `doc/EXTERNAL_DESIGN.ja.md` と実装が矛盾している
- テストが期待する値と既存文書が矛盾している
- option / field は存在するが、許可値や不正値時の挙動が不明
- ラベル (§2.8) を `stable` と確定させてよいか判断できない

未確定事項として残す各項目には、次を併記する。

- 確認すべき対象ファイル (実装コード / artifact / test)
- 確認手順または確認コマンド (例: `spec-anchor core --rebuild` を実行して `freshness_report.diagnostics` を観察)
- 決めるべき契約内容 (何を確定させれば断定値に昇格するか)
- 関連する既存記述 (どの章・どの行と整合させるか)
- 暫定的に設定するラベル (§2.8、確定前は `experimental` 扱い)

「未確定事項」節は、契約の網羅性を犠牲にせず推測も避けるための逃げ道である。未確定のままで仕様書を公開してよい。未確定のまま放置しないために、項目には確認コマンドと判断要件を必ず添える。

## 5. 外部仕様書の構成

```
1. 外部仕様の対象範囲
2. 用語
3. 共通実行規約
4. 設定ファイルの外部契約
5. /spec-core の外部仕様
6. /spec-inject の外部仕様
7. /spec-realign の外部仕様
8. spec-anchor-watch の外部仕様
9. spec-anchor-setup-system の外部仕様
10. spec-anchor-setup-project の外部仕様
11. Conflict Review Item の外部仕様
12. freshness gate の外部仕様
13. エラー仕様
14. 互換性と非保証事項
```

Appendix セクションは設けない。`spec-anchor-setup-project` が生成する初期 `.spec-anchor/config.toml` 全文は §4 設定ファイルの外部契約章の中に掲載する。Agent / 利用者 / 後続コマンドが参照する artifact JSON の field 一覧は、それぞれを生成するコマンド章 (§5〜§10) または §11 / §12 / §13 の該当章内に分散して掲載する。一覧専用の Appendix を別途置かない理由は、本文と Appendix の二重管理を避けるためである。

## 6. 各コマンド章の必須フォーマット

SPEC-anchor のコマンド契約は次の 2 層に分かれる。コマンドによって両方の層が存在する場合と片方だけの場合がある。

- **shell CLI 契約**: 利用者または Agent が `spec-anchor` / `spec-anchor-watch` / `spec-anchor-setup-*` を子プロセスとして実行したときの入出力契約
- **Agent slash command / skill 契約**: 利用者が Claude Code の slash command または Codex の skill を発火し、Agent CLI が内部で `spec-anchor` を呼んで利用者へ最終出力を返すまでの契約

各コマンド章では、該当する層をすべて書く。

| コマンド | shell CLI 契約 | Agent slash command / skill 契約 |
|---|---|---|
| `/spec-core` (内部実行: `spec-anchor core`) | 必須 | 必須 |
| `/spec-inject` (内部実行: `spec-anchor inject-*`) | 必須 | 必須 |
| `/spec-realign` (内部実行: `spec-anchor realign`) | 必須 | 必須 |
| `spec-anchor-watch` | 必須 | 該当なし (slash command 経由で発火しない) |
| `spec-anchor-setup-system` | 必須 | 該当なし |
| `spec-anchor-setup-project` | 必須 | 該当なし |

### 6.1 shell CLI 契約の必須節

shell CLI 契約は次の節をすべて書く。節を省略しない。該当 0 件の場合も「該当なし」を明示する。「主なオプション」のような選別表現は使わず、サポートする全項目を列挙する。

#### 実行形式

CLI 実行コマンドラインの基本形を 1 つ示す。

#### 入力

| 入力 | 必須 | 説明 | 不備がある場合の挙動 |

#### オプション

| オプション | 必須 | 既定値 | 意味 | 外部から観測できる効果 |

#### 出力

| 出力項目 | 型 / 形式 | 必須 | 意味 |

stdout JSON は top-level field と、Agent / 利用者 / 後続コマンドが参照する nested field を網羅する。

#### 作成・更新される artifact

| artifact | 作成条件 | 更新条件 | 更新しない条件 |

#### 正常終了

- CLI exit code
- stdout JSON の主要 field
- stderr に出る情報
- 後続コマンドが実行可能か

#### 警告付き続行 (該当する場合)

- 発火条件
- CLI exit code
- stdout JSON に追加される warning field の値
- 後続コマンドが実行可能か

#### 停止 / 失敗

| 発火条件 | reason | 利用者に表示される内容 | CLI exit code | 利用者の次アクション |

#### 副作用として行わないこと

negative contract を列挙する。

### 6.2 Agent slash command / skill 契約の必須節

slash command 契約は、Agent CLI (Claude Code / Codex) が利用者の発火を受け、内部で `spec-anchor` shell CLI を呼び、その JSON 戻り値を読んで利用者へ最終出力を返すまでの契約である。次の節をすべて書く。

#### 発火形式

利用者が Agent CLI で入力する slash command または skill 呼び出しの形 (`/spec-inject "<task>"` など)。

#### Agent が内部で呼び出す shell CLI

shell CLI 契約 (§6.1) のどの実行コマンドラインを発火するかを列挙する。

#### Agent が利用者へ提示する正常時の出力

利用者宛に Agent が提示する最終セクション構造 (見出し、必須セクション、各セクションに含めるべき情報)。Agent が自由文で要約する範囲と、shell CLI JSON から転写する範囲を分けて書く。

#### Agent が停止時に利用者へ提示する内容

| 停止理由 (shell CLI の `blocking_reasons` 等) | 利用者宛 Agent 出力に必ず含める情報 | Agent が提案する次アクション |

#### Agent が追加確認を求める条件

Agent が利用者へ「課題が特定できない」「answer candidate が欲しい」「人間判断を返してほしい」等の追加情報を要求する条件と、その表示内容。

#### shell CLI exit code / status の Agent 出力への写し方

shell CLI が `status=failed` / `status=blocked` / `status=error` / exit code 1 を返したとき、Agent が利用者へどう伝達するかの規約。raw JSON を貼らず構造化して伝える、`recommended_next_action` を必ず転写する、など。

#### Agent slash command 層で行わないこと

Agent CLI が自由生成で shell CLI の契約 (status、reason、artifact path、constraint structure など) を書き換えて利用者へ伝達してはいけないこと、shell CLI を呼ばずに会話だけで「実行した」と装ってはいけないこと、などの negative contract を列挙する。

## 7. 設定ファイル章の必須フォーマット

`.spec-anchor/config.toml` の全 table / 全 key を次の table 形式で列挙する。

| Table | Key | 型 | 必須 | 既定値 | 許可値 | 不正値時のエラー | 外部から観測できる効果 |

「内部 tuning なので省略」「Agent / 利用者は通常触らないので省略」という扱いはしない。`[llm.providers]` / `[retrieval]` / `[embedding]` / `[vector_store]` / `[limits]` / `[watcher]` も同じ table に全 key を載せる。各キーが内部でどう使われるかは内部設計に書き、外部仕様には観測効果までを書く。

`spec-anchor-setup-project` が初期化時に展開する初期設定 TOML は、内容が利用者の目に触れる契約物である。§4 設定ファイル契約章の中に、全 table / 全 key の仕様表と、初期 TOML の全文をどちらも掲載する。Appendix などの別セクションには切り出さない。仕様表と初期 TOML 全文は同じ章内で同期する。

## 8. 文体ルール

- 1 文を短くする
- 先に結論を書く
- 内部用語を本文で使うときは、利用者・運用者・Agent が実際に入力・確認するものに限る (CLI flag、設定キー、artifact path、識別子、status 値、reason code など)
- 「〜しない」だけで終わらず、先に「〜する」を書く
- 表のセルで「あれば」「など」のような曖昧化を避ける
- 「主なオプション」「正常時に見る結果」のような選別表現で終わらせない。stdout の field、exit code、artifact 更新、後続コマンド可否まで全部書く
- 検証進捗マーク (`[ ]` / `✅`) を本文に埋め込まない

## 9. 既存資料との関係

### 既存 `doc/EXTERNAL_DESIGN.ja.md`

編集しない。本指針は既存ファイルを書き換える指示ではない。既存ファイルはテスト依存があるため、本指針に従って作成する新ドラフトとは別管理。

### 新ドラフトファイル名

`doc/EXTERNAL_SPEC_DRAFT.ja.md` として新規作成する。`DRAFT` 接尾辞は「これは試作段階であり、`doc/EXTERNAL_DESIGN.ja.md` を置き換える正本ではない」ことを示す。Agent / 利用者の参照経路や CLAUDE.md / AGENTS.md / TODO 上での正本扱いは、ドラフト完成と人間判断による正本切り替え合意の後に別途決定する。

### `doc/README素材.md`

## 10. 前回失敗の禁止リスト

次のいずれかが本文に発生したら、外部仕様書として失敗とみなす。書き直す。

- 利用者ガイド / Getting Started の文体に流れる
- 「人間に読みやすくするため」を理由に契約情報 (CLI option、stdout field、exit code、状態遷移、error reason code、artifact 更新条件、設定キー) を削る
- 「主なオプション」「主な使い方」のような選別表現で全列挙を回避する
- 検証進捗マーク (`[ ]` / `✅`) を本文に埋め込む
- 「Qdrant」「embedding」「payload」「UUID5」「fingerprint」「cache」「collection」を理由なく排除して、Agent / 利用者が打鍵する識別子や、利用者が編集する設定キーまで削る
- 設定キーを「内部 tuning」とラベルして本文から省略する
- 状態名 / status 値 / decision 値 / reason code を抽象表現 ("解決済み" "保留" など) に置き換えて、機械可読 enum を本文から消す
- artifact path を「保持物」のような抽象表現に統合して、個別 path を本文から消す
- Conflict Review Item の `valid_scope` などの契約値を文章で要約して enum を消す

これらは「読みやすさのため」の名目で発生しやすい。読みやすさよりも契約の網羅性と正確性を優先する。

「人間が読める外部契約書」とは、「ソースコードを読まなくても、機械可読な契約 (option / field / status / reason / artifact path / 設定キー) を網羅して理解できるように書かれた仕様書」のことである。
