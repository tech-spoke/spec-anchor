# 外部設計書の記述ルール

本文書は、`doc/EXTERNAL_DESIGN.ja.md` を記述・レビューする際のルールを定める。

外部設計は、ソースコード、内部アーキテクチャ、クラス名、モジュール名、グラフスキーマ、キャッシュ構造、実装メモを読んでいない読者にも理解できなければならない。

外部設計に含めてよい文は、次のうち少なくとも 1 つに答えている場合に限る。

1. ユーザーは何ができるのか？
2. ユーザーは何を観測できるのか？
3. システムはどのような保証を提供するのか？
4. ユーザーが知っておくべき制限は何か？
5. その挙動が発生したとき、ユーザーはどのような操作を取るべきか？

次の知識がないと理解できない文は、外部設計に含めてはならない。

- 内部モジュール
- 内部データ構造
- グラフのノード種別またはエッジ種別
- キャッシュ実装
- 分類パイプラインの内部処理
- プロバイダー固有の実装
- ソースコード上の名前
- テスト専用プロファイル
- デバッグ専用の用語

その情報が有用な場合は、内部設計 (`doc/DESIGN.ja.md`) または実装メモへ移動すること。

## 情報密度のルール (CLAUDE.md ルール 14)

仕様書の記述は「定義文から読者が自力で導けない情報」だけを書く。定義文で確定している内容を繰り返さない。

判定基準: **定義文を読んだ読者がこの行を自力で導出できるか?**

- 導出できる → 書かない (認知コストだけ上がる)
- 導出できない → 書く (もれなく記述すべき独自情報)

近隣セクションの形式 (含む / 含まない テンプレート等) を機械的に踏襲して、自明な内容を列挙してはいけない。

## ソースコード未読の読者テスト

外部設計の文を書く前に、Agent は次を確認しなければならない。

「この文は、ソースコードを読んでいないユーザーにも理解できるか？」

答えが no の場合、Agent は次のいずれかを行わなければならない。

1. ユーザーが観測できる言葉で文を書き直す
2. その文を内部設計へ移動する

悪い例:

> `classification_incomplete` は、source / concept / chapter / cluster の候補数が分類予算を超えたときに発生する。

良い例:

> システムが安全に分類できる数を超える根拠候補を持つ場合、回答が不完全である可能性を警告する。
> ユーザーは、その回答を暫定的なものとして扱い、信頼して利用する前に refresh コマンドを実行するべきである。

## 必要性の分類

内容が必要かどうかを判断するとき、Agent は必要性を次のいずれかに分類しなければならない。

- 外部設計に必要
- 内部設計に必要
- 実装メモに必要
- テストに必要
- 不要

Agent は、単に「必要」または「不要」とだけ答えてはならない。
